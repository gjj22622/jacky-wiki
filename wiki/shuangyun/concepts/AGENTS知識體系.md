## 1. 為什麼需要 AGENTS® 方法論

第一，**行銷人可以成為 AI 架構師**。不需要寫程式，只需要把工作邏輯講清楚。
第二，**方法比工具重要**。工具會換（Gemini / ChatGPT / Claude / NotebookLM），方法論不會過時。
第三，**零散的知識需要體系化**。四週實戰累積的方法論碎片（流程拆解表、九區塊設定檔、三色殘酷測試、三種協作模式、Skill 瘦身術）需要一個統一框架把它們串起來。

> AGENTS® 是一個自指式縮寫（self-referential acronym）：用 AGENTS 六個字母，定義建構 AI Agent 系統的六個階段。

### Jacky 的方法論作品脈絡

SOSTAC®（國際引進，Jacky 學習與整合 14 年）

> AGENTS® 與 [SPEAKS®](../../tbsa/concepts/SPEAKS方法論總覽.md) 都是 Jacky 從 SOSTAC 模型「產出」的方法論——前者解決「會建（AI Agent）」，後者解決「會說會教（口說表達）」。

## 2. AGENTS® 六階段總覽

| 階段 | 字母 | 英文 | 中文核心 | 一句話定義 | 對應週次 |
|:---:|:---:|---|:---:|---|:---:|
| 1 | **A** | Analyze | 分析拆解 | 把人的工作拆成 AI 能處理的步驟 | Week 1 |
| 2 | **G** | Generate | 生成建構 | 把每個步驟轉化為獨立的 Agent 設定檔 | Week 2 |
| 3 | **E** | Evaluate | 評測修正 | 用三色情境殘酷測試 Agent 的真實能力 | Week 3 |
| 4 | **N** | Network | 串聯協作 | 把多個 Agent 連接成工廠級工作流 | Week 4 |
| 5 | **T** | Track | 追蹤管理 | 用 API 串接實現自動化追蹤與品質管控 | Week 5-6 |
| 6 | **S** | Scale | 規模交付 | 開放客戶端 API、建立自動進化機制 | Week 7-8 |

### 六字訣映射

```
拆 → 建 → 修 → 串 → 管 → 交
A      G      E      N      T      S
```

> 前四階段（A-G-E-N）是「基礎篇」，任何人都能在四週內完成。
> 後兩階段（T-S）是「進階篇」，需要技術架構支持。

### 三個不變原則

1. **人定義邏輯，AI 執行重複** — 人的價值在判斷和決策，AI 在不疲勞地重複執行
2. **一步一 Agent，規格即合約** — 每個 Agent 只做一件事，靠嚴格的交接規格溝通
3. **測試驅動信任** — Agent 的可靠性不靠 System Prompt 長度決定，靠通過多少殘酷測試決定

## 3. 六階段詳細定義

### A — Analyze（分析拆解）

> **核心問題**：我的工作裡，哪些步驟可以交給 AI？
> **產出物**：工作流拆解表

**六維度拆解表（權重）**

| 維度 | 權重 | 內容 |
|---|:---:|---|
| 基本資訊 | 15% | 流程名稱、觸發條件、最終產出（流程身分證） |
| 步驟拆解 | 25% | 3-10 步驟，每步「一個動作 + 一個驗證方式」 |
| 判斷邏輯 | 25% | if-then 規則（Agent 的大腦） |
| 資料來源 | 15% | Dataroom 結構（沒有資料的 Agent 只能幻想） |
| AI 適配評估 | 10% | 全自動／半自動／純人工 |
| 眉角紀錄 | 10% | 師父腦中不會寫在 SOP 裡的隱性知識 |

**驗收標準**：🌟 80%+ 可直接轉 Agent ／ ⭐ 60-79% 需補完 ／ 🔴 <60% 需 1-on-1 引導

**三大常見地雷**

1. 步驟太粗（「處理客戶需求」要拆成 4 步：收集→確認→推薦→同意）
2. 判斷邏輯藏在直覺裡（「看情況決定」不是判斷邏輯）
3. 忘了標記資料來源

---

### G — Generate（生成建構）

> **核心問題**：怎麼把一個步驟變成一個能運作的 Agent？
> **產出物**：Agent 設定檔 ×3 以上

**九區塊設定檔**

1. 角色定位（一句話說明 Agent 是誰）
2. 觸發條件（什麼情況啟動）
3. 輸入需求（必填／選填欄位）
4. 執行流程（步驟＋動作＋決策點＋產出）
5. 判斷邏輯與決策樹（>10 條規則 → 需要瘦身）
6. **輸出規格**（最關鍵：本 Agent 的輸出 = 下一個 Agent 的輸入）
7. 知識庫／Dataroom 需求
8. 邊界與禁止行為（遇到什麼要停下交回人類）
9. 進化機制（品管回饋紀錄、Prompt 版本控制）

**一步一 Agent 三規則**

1. **單一決策** — 一個步驟兩個判斷點 → 拆兩個 Agent
2. **格式相容** — A 的輸出格式 = B 的輸入格式
3. **獨立可測** — 不依賴上游 Agent 即時回應

**跨平台部署原則**

| 平台 | 角色 | 適合場景 |
|---|---|---|
| Claude Skill | 深度分析、複雜工作流 | 主力執行平台 |
| Gemini Gem | 搜尋整合、Gmail / Calendar | Google 生態整合 |
| ChatGPT GPTs | 類似 Gem | OpenAI 生態 |
| NotebookLM | 知識庫查詢 | 唯讀型、不執行 |

**驗收標準**：每位學員至少 3 個 Agent，每個至少 1 個平台部署成功並通過 3 個測試情境（標準／邊緣／錯誤）。

---

### E — Evaluate（評測修正）

> **核心問題**：Agent 在極端情況下會不會出包？
> **產出物**：殘酷測試報告 + 修正後的 Agent v2
> **核心信念**：**沒經過殘酷測試之前，Agent 只是玩具不是工具。**

**三色情境**

- 🟢 **標準**：完整正確的輸入，測基本功能
- 🟡 **邊緣**：合法但棘手（同時符合兩方案、預算剛好在門檻、需求模糊）
- 🔴 **錯誤**：故意錯誤（假數據、矛盾資訊、AI 味過重、法規違規）

**三階段測試流程**

| Phase | 名稱 | 動作 |
|---|---|---|
| A | 自測 | 同一情境各測各的，結果上傳共用資料夾比較 |
| B | 交叉測試 | 兩兩配對用邊緣情境測對方的 Agent |
| C | 紅色測試 | 全班用刻意錯誤的資料攻擊（90% 學習在這） |

**修正工作坊** — 測完不修等於白測。當學員發現某 Agent 因做太多事而失敗 → 自然引入「Sub Agent」概念。

**驗收標準**

| 指標 | 及格 | 優良 | 卓越 |
|---|:---:|:---:|:---:|
| 🟢 標準通過率 | 100% | 100% | 100% |
| 🟡 邊緣通過率 | ≥60% | ≥80% | ≥90% |
| 🔴 錯誤攔截率 | ≥50% | ≥70% | ≥85% |
| 交叉測試回饋 | ✓ | ✓ | ✓ |
| Before/After 修正紀錄 | ≥1 | ≥2 | ≥3 |

---

### N — Network（串聯協作）

> **核心問題**：怎麼讓多個 Agent 一起工作，變成一座工廠？
> **產出物**：Agent 工作流設計圖 + Skill 瘦身盤點表

**三種協作模式**

| 模式 | 說明 | 適合 | 速度 | 風險 |
|---|---|---|:---:|---|
| Pipeline | A → B → C 串行 | 有固定順序的流程 | 最慢 | 任何節點格式錯誤 → 斷鏈 |
| Sub Agent | 主 Agent 呼叫專業判斷 | 專業環節（合規、法規） | 中等 | Sub 沒回覆需 fallback |
| Parallel | 多 Agent 同時工作後整合 | 任務彼此獨立 | 最快 | 整合端品管壓力大 |

**Sub Agent 拆分四訊號**

1. Agent 做太多事（判斷規則 > 10）
2. 需要專業知識（法規、金融、醫療）
3. 複用需求出現（≥ 2 Agent 都需要同功能）
4. 品質下降（System Prompt > 2000 字，開始失焦）

**Skill 瘦身術**

| 指標 | 健康 | 需要瘦身 | 必須立刻瘦身 |
|---|:---:|:---:|:---:|
| System Prompt 長度 | <1000 字 | 1000-2000 字 | >2000 字 |
| 判斷規則數量 | <5 條 | 5-10 條 | >10 條 |
| 殘酷測試通過率 | >80% | 60-80% | <60% |

三步驟：**盤點 → 拆分 → 連接**

**交接規格書（Handoff Specification）** — 工廠運作的關鍵不是每個 Agent 多厲害，而是 Agent 之間的交接有多精確。包含：傳出方欄位、接收方期待、錯誤處理（格式不對／超時／品質不及格怎麼辦）。

**驗收標準**：產出完整的工作流設計圖（含 Pipeline / Sub / Parallel 標記、交接規格、錯誤處理）+ Skill 瘦身盤點表。

---

### T — Track（追蹤管理）

> **核心問題**：怎麼讓 Agent 工廠自動跑起來，並確保品質持續達標？
> **產出物**：自動化流水線 + API 調度系統 + 品質監控儀表板

**API 調度中心 — 四層架構**

| 層 | 功能 | 工具 |
|---|---|---|
| 觸發層 | 接收指令 / 排程 / 客戶請求 | n8n Form / LINE Bot / Webhook |
| 編排層 | Main Agent 路由、流程控制 | n8n Workflow + 路由邏輯 |
| 執行層 | 各 Sub Agent 執行專業任務 | Codex API + System Prompt |
| 儲存層 | Dataroom、進度、日誌 | Google Drive / Notion |

**讀寫分離原則**

- **GET（讀）**：不產生新內容，只讀 Dataroom；不需品管、不存檔、可大量快取
- **POST（寫）**：創造新內容、改變狀態；必須品管、記錄日誌、關鍵節點人確認

**品質管控機制**

1. **自動品管**：每次寫操作後，品管 Agent 自動給分（1-10）+ 修改原因
2. **進化日誌**：自動彙整最近 10 次品管回饋；累積 5+ 次相同類型修改 → 自動產出 Prompt 修改建議
3. **人工確認**：所有 Prompt 變更必須 Jacky（或指定管理者）確認，**Agent 不允許自行修改自己的 System Prompt**

> 完整路由表（9 GET + 8 POST）見 [`AGENTS方法論落地.md` §4.6](AGENTS方法論落地.md)。

**驗收標準**：至少一條完整自動化流水線跑通，品管 Agent 能自動攔截不合格產出，Trace Log 完整。

---

### S — Scale（規模交付）

> **核心問題**：怎麼讓客戶直接用我們的 Agent 工廠，並讓系統越用越好？
> **產出物**：客戶 API 端點 + 營運儀表板 + 自動進化機制

**三層開放架構**

- **認證層**：每客戶獨立 API Key、速率限制、Dataroom 隔離
- **計費層**：基本月費（NT$8,000）+ 超出篇數費（NT$100/額外篇）；用量即時追蹤、月底自動結算
- **監控層**：營運儀表板顯示用量、品管分數趨勢、執行成功率、平均回應時間、月成本

**Agent 自動進化**

1. **數據驅動**：品管 Agent 的每次評分都是訓練資料；某類錯誤反覆出現 → 自動產出 Prompt 修改建議
2. **版本控制**：每次 Prompt 修改有版本號 + 修改原因 + 前後對比；可回滾任何歷史版本
3. **A/B 測試**：新版 Prompt 先對 10% 請求生效，確認品質提升後全面推送

**驗收標準**：至少一個外部客戶透過 API 完成完整內容產出；儀表板即時顯示關鍵指標；自動進化機制至少產出過一次有效改善建議。

## 4. 三級認證體系

> 操作細節（評分表、評分標準、考核流程）見 [`AGENTS認證手冊.md`](AGENTS認證手冊.md)。

| 等級 | 名稱 | 對應階段 | 核心能力 | 取得方式 |
|---|---|---|---|---|
| **L1** | Agent Builder | A + G + E | 把工作流程拆 → 寫 Agent → 通過殘酷測試 | Week 1-3 課程 + 完整作業 + 評分 Agent 審核 |
| **L2** | Agent Architect | A + G + E + N | 設計多 Agent 協作工作流 | Week 1-4 課程 + 工作流實戰 + 口頭 Demo |
| **L3** | Agent Factory Manager | 全六階段 | 搭建並維運企業級 Agent 工廠 | Week 1-8 課程 + 實戰專案 + Jacky 面審 |

### 四維驗證法（佔比隨等級調整）

| Level | 知識考核 | 實作產出 | 實戰應用 | 同儕認可 |
|---|:---:|:---:|:---:|:---:|
| L1 | 25% | 35% | 25% | 15% |
| L2 | 20% | 30% | 30% | 20% |
| L3 | 15% | 25% | 35% | 25% |

> **越高等級，「實戰」和「同儕認可」佔比越高**——到工廠經理層級，「做出來別人真的能用」比「自己懂理論」更重要。

### 認證有效期

| 等級 | 有效期 | 續證條件 |
|---|:---:|---|
| L1 | 1 年 | 過去一年至少 5 次 Agent 使用紀錄 |
| L2 | 1 年 | 過去一年至少 1 條運作中的工作流 + 進化紀錄 |
| L3 | 2 年 | 過去兩年的營運數據 + 至少 1 次系統升級紀錄 |

到期前 30 天自動提醒，到期後 90 天內可補件續證，超過 90 天需重新考核。

## 6. 三場景部署

### 6.1 場景一：双云內部（已驗證）

- **對象**：7 位行銷夥伴

- **認證路徑**：全員至少達 L2，Jacky 達 L3

**已驗證的教學節奏**

| 週次 | AGENTS 階段 | 課堂 | 課後作業 | 驗收點 |
|:---:|:---:|:---:|---|---|

| 2 | G 生成建構 | 120 min | 3+ 設定檔 | 至少 1 平台部署 |

### 6.2 場景二：企業客戶培訓（可複製）

- **對象**：企業內部團隊（5-15 人）
- **週期**：4-8 週（依需求）

**標準化交付套件**：課程簡報、六維度／九區塊模板、殘酷測試情境庫、評分 Agent、認證考核表、課後追蹤儀表板

**客製化要素**：A 階段「拆什麼」、G 階段 Agent 範例、E 階段產業專屬風險

**定價建議**

| 方案 | 內容 | 建議價格 |
|---|---|---|
| 基礎班（A-G-E） | 3 週 + L1 認證 | NT$150,000–250,000 |
| 進階班（A-G-E-N） | 4 週 + L2 認證 | NT$250,000–400,000 |
| 完整班（全六階段） | 8 週 + L3 認證 | NT$500,000–800,000 |

## 7. AGENTS® 與其他方法論的整合

```
┌──────────────────────────────────────────────────┐
│                Jacky 知識體系全貌                   │
│                                                    │
│  SOSTAC®          SPEAKS®          AGENTS®         │
│  行銷企劃          口說表達          AI Agent        │
│  方法論            方法論           方法論           │
│  ──────          ──────          ──────           │
│  會想+會做         會說             會建             │
│                                                    │
│  S-O-S-T-A-C     S-P-E-A-K-S     A-G-E-N-T-S     │
│                                                    │
│       策略層 ─────────────→ 執行層                  │
│       SOSTAC 的 A（行動）   AGENTS 的 A（分析）      │
│       SOSTAC 的 C（管控）   AGENTS 的 T（追蹤）      │
│                                                    │
│       表達層 ←────────────→ 萃取層                  │
│       SPEAKS 的 S（掃描）   AGENTS 的 A（分析）      │
│       把隱性知識說出來       把隱性知識拆給 AI        │
└──────────────────────────────────────────────────┘
```

### 與 SOSTAC® 的整合點

SOSTAC® 是策略層，AGENTS® 是執行層。
- SOSTAC® 的 **A**（Action）階段產出的行動計畫 = AGENTS® 的 **A**（Analyze）階段要拆解的工作流程
- SOSTAC® 的 **C**（Control）階段需要的 KPI 追蹤 = AGENTS® 的 **T**（Track）階段要建的監控系統

### 與 SPEAKS® 的整合點

兩者在「知識萃取」交會——AGENTS® 的 A 階段需要把隱性知識「問出來」，這就是 SPEAKS® 的 **S**（Scan）能力。

|---|---|

## 9. 名詞速查

| 名詞 | 定義 |
|---|---|
| Agent | 具備特定角色、觸發條件、輸入輸出規格的 AI 執行單元 |
| Sub Agent | 被主 Agent 呼叫的專業功能單元 |
| Skill | Agent 的設定檔封裝格式（System Prompt + 參考資料 + 腳本） |
| Dataroom | Agent 可存取的結構化知識庫（Agent 的記憶體） |
| 交接規格 | 兩個 Agent 之間傳遞資料的格式定義（工廠運作的合約） |
| System Prompt | Agent 的核心指令 |
| Pipeline / Parallel | 串行 / 並行工作流 |
| 殘酷測試 | 三色情境（標準／邊緣／錯誤）刻意壓力測試 |
| Skill 瘦身 | Agent 過載時的功能拆分流程 |
| 讀寫分離 | API 設計原則（GET 不品管 / POST 必品管） |
| 進化機制 | Agent 基於品管回饋持續改善 System Prompt |
