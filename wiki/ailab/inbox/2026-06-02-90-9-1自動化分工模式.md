## 發生了什麼

整套「收信→建客戶→回信」自動化做完後，浮現一個 Jacky 認可的分工比例：
**90% 確定性程式（不用開機/不用開對話也自己跑）＋ 9% LLM 在重要節點判斷 ＋ 1% 人工控制敏感節點。**
關鍵心法是：**LLM 的角色是「寫好確定性程式」，而不是「當 runtime 一直在跑」**——讓絕大多數工作免 LLM 持續運作。

## 為什麼重要

打破「全自動 = 養一個常駐 LLM 一直讀一直做」的迷思。常駐 LLM 貴、慢、會幻覺、要有人盯著開機。
這個 90/9/1 分工同時拿到三個好處：**省 token（90% 不呼叫 LLM）、低出錯（確定性程式不幻覺）、不需常駐 AI（cron/daemon 24h 自走）**，而 LLM 只在「真的需要判斷」的節點被確定性程式按需呼叫，敏感的事留給人。
今天它不是理論——是被真實流量驗證的：林芭比報名時系統秒級自動建檔+回信（90% 那層），LLM 自動回信對客訴/報價會轉人（9%→1% 那層），且 LLM 報告誤洩內部稽核資料當場被人工閘門攔下（1% 的必要性反證）。

## 怎麼做的

今天這套 harness 就是 90/9/1 的實體：

- **90% 確定性（無 LLM、systemd/cron 24h）**：`gmail-watch.py`(Gmail History API 收信→Telegram)、`mail-poll.js`(cron 5min 寫 CRM)、`customers.js ensureCustomerByEmail`(email→客戶UUID，雜訊用 regex 過濾、決定性 slug)、確定性 auto-ack(報名信秒回「收到」固定模板)、`cron-watchdog`。這些都是固定程式，不開 Claude 對話也持續運作。
- **9% LLM 判斷（確定性程式按需呼叫 `claude -p` headless）**：`reply-agent.js`——客戶第 2 封起的回信交給 LLM 讀內容、接手需求、產回信；體檢產線 `airun-venture-checkup`(SOSTAC 五表+對抗式體檢)。LLM 不常駐，是被 daemon 在「判斷節點」spawn 出來、用完就退。
- **1% 人工敏感閘門**：reply-agent 的「自動寄+敏感轉你」——LLM 自判敏感(報價承諾/客訴/退費/法律/不確定)就不寄、推 Telegram 等 Jacky；對外發信凍結/解凍；報告品質校準(這次靠人工抓到 LLM 把內部稽核黑歷史洩漏進客戶報告)。

口訣：**「LLM 建管線、LLM 守判斷點、人守敏感閘」。**

## 對比與替代

- **全自動常駐 LLM**：曾出包——AIRUN 早期「全自動」outreach 偽造「已發送」狀態、$16.5/日心跳費用失控（73% 幻覺事故那條線）。證明把判斷無上限交給常駐 LLM 會幻覺+燒錢。
- **全人工**：慢、不可規模化、要一直開機開對話。
- **90/9/1 中道**：確定性程式扛掉重複勞動，LLM 只在判斷節點出現，人只在「寄出去不可逆/牽涉金錢法律」的 1% 出現。今天的洩漏事故與客訴轉人，都證明 1% 人工閘門不能省。
