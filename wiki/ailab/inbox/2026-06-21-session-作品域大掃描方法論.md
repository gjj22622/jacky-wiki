## 最終做法（What Worked）

**整套「資產盤點歸檔法」——下次掃任何一批散亂作品都照這做：**

1. **先輕量盤點頂層**（主控自己跑 `ls` + 訊號偵測：.git / package.json / Dockerfile / *.html / 最後修改日），拿到候選清單才分派 agent。
2. **派 agent team 並行調查**（Explore，read-only），每 agent 查一批夾，回報**結構化座標**：定位 / git remote / 前端網址 / admin / 部署平台 / DNS / 狀態 / 憑證檔是否存在。
3. **寫入集中由主控做**（agent 只調查不寫）——這是守金鑰紅線的關鍵：agent 不碰寫入、不貼 token 值。
4. **深掘鐵則**：資料夾「子夾 >5 個」或「深度 >3 層」→ 派**專屬 agent** 深掘那一夾。大夾常是「**專案家族**」內含多個獨立子作品，逐一揪出。
5. **作品定義放寬**：①已部署/有網址 ②AI 生成簡報/素材 ③有導覽頁/HTML，符合任一即收。
6. **分離金庫**：`wiki/portfolio/.vault/` 進 `.gitignore`；頁面只記「帳號→見金庫」+ 憑證檔路徑（指標式），token 值只進金庫。每次 commit 前 `git check-ignore` 驗證金庫沒外洩。
7. **分類目錄 A-E + 專案家族巢狀**：量大時分類比 N 個散頁好檢閱；旗艦/已部署給完整頁（座標+登入見金庫+一鍵示範腳本），其餘單行條目。
8. **交叉比對**：OneDrive 工作副本 vs 本機 git 源，判斷同一個避免重複歸檔。
9. **本機 + OneDrive 兩處都要掃**（素材源頭分散）。
10. 沉澱成**可交接 SOP**（`portfolio/作品歸檔SOP-協作者交接.md`）給 J董 等協作者複製貼上跑同一套。

## 錯誤與失敗（What Failed）

- **把 Jwood 誤標「無 git 備份風險」** → Jwood 是**物理世界的木藝創作作品**（綠檀按摩棒），實體創作沒有 git 是天經地義、不是缺漏。被 Jacky 訂正後正名：作品本體=實體木藝，OneDrive 夾只是支援品牌的內容工作流工具。**教訓：別用軟體框架去套實體作品。**
- **盲目信任 agent 的表層回報** → Explore agent 預設不深入子夾，主控要明確下「深掘 N 層、逐一找子專案」指令才挖得到。
