# 双云 AI 行銷部 — 定義與操作指南

## Part A — 為什麼要建 AI 行銷部

### A.1 我們在解決的問題：行銷流程的摩擦力

每一個行銷部都有同樣的病：**腦跟手不同步。**

老闆的想法，在傳達給執行的過程中，不斷被稀釋、被誤解、被遺忘。

```
老闆想：「要有質感、要打動媽媽、不要太商業」
  ↓  （摩擦力發生）
執行產出：土味促銷文、過度賣貨、跟上週風格完全不同
```

這不是執行者的錯。是**指揮系統本身有摩擦**。

### A.2 摩擦力的四個來源

| # | 摩擦點 | 常見症狀 |
|---|---|---|
| ① | **老闆意志傳達不準** | 說「有質感」→ 出土味；說「不要賣貨」→ 每篇都在促銷 |
| ② | **策略與執行脫節** | 月曆定好了 → 執行沒看；KPI 說衝 ROAS → 貼文沒行動 |
| ③ | **執行細節被遺忘** | 客戶說不要用「！！！」→ 每篇都有；上次調整的語氣 → 下次又跑掉 |
| ④ | **品牌一致性崩潰** | 換了一個執行者 → 品牌感覺整個不同；本週跟上週像兩個品牌 |

> **只要有一個沒解決，整個行銷部就在漏電。**

## Part B — 4 Agent 委員會（每個客戶一套）

每一個摩擦點，對應一個 Agent。**四個 Agent，缺一不可。**

| # | 摩擦點 | 對應 Agent | 守護內容 |
|---|---|---|---|
| ① | 老闆意志傳達不準 | 🔴 **老闆 Agent** | 品牌願景、絕對禁忌、重大決策否決權 |
| ② | 策略與執行脫節 | 🟡 **主管 Agent** | KPI 方向、策略一致性、行銷計畫對齊 |
| ③ | 執行細節被遺忘 | 🟢 **窗口 Agent** | 客戶偏好、語氣禁忌、溝通習慣細節 |
| ④ | 品牌一致性崩潰 | 🔵 **品牌 Agent** | CI 規範、視覺語調、核准例句庫 |

每個客戶都有自己獨立的 4 Agent 委員會 + Google Form 資料庫。但 **6 個跨客戶共用的「手 Skill」**（產文／圖／影片腳本／企劃／廣告／週報）由 Jacky 維護，不需要重複建。

## Part C — 老闆做什麼，AI 做什麼

> 整個 AI 行銷部的靈魂：**清楚的分界線。**

### 老闆做的事（只有老闆能決定）— 1-4

```
1. 找問題   「我們現在最痛的是什麼？」
2. 定邊界   「這個做，那個不做，這個碰了就死」
3. 設目標   「這季要達到什麼結果？用什麼衡量？」
4. 給方向   「用這個策略，不用那個」
```

老闆說完，就放手。**後面的事不需要管。**

### AI 行銷部做的事（自動接手）— 5-8

```
5. 想策略   依據老闆方向，拆解成行銷計畫與內容方向
6. 排執行   貼文、EDM、廣告策略、週報 — 自動產出
7. 自品管   4 Agent 委員會審核，不合格自動退回重做（最多 2 輪）
8. 學回饋   客戶回饋回流，腦持續進化，下次更準
```

### 鐵律

> 老闆只輸入 1-4，不問「你喜歡哪個版本」。
> AI 行銷部只做 5-8，不打擾老闆確認每一篇貼文。

## Part D — 「串」是什麼

「串」= 腦跟手之間的絲滑連結。當每一個輸出，自動成為下一個的輸入，就叫「串起來了」。

```
老闆說一句話
    ↓ 串
腦層理解方向，產出策略包
    ↓ 串
手層接收策略包，產出內容
    ↓ 串
4 Agent 委員會審核，PASS 或退回重做
    ↓ 串
結果自動寫入品牌資料庫（Google Form）
    ↓
老闆看結果，給下一個方向
```

**沒串好的地方 = 摩擦點 = 需要人工補位的地方。**

目標：把你手上的每個品牌，所有摩擦點都串起來。

### Phase 1｜流程拆解（人 + AI 協作）

把每天在做的工作，拆成 AI 能理解的步驟。**不是寫 SOP，是用「教新人」的方式講出來。**

#### 三種拆解方法

- **方法 A — 自己寫**（適合內向型、已經很清楚流程的人）：打開模板逐欄填寫
- **方法 B — 跟 AI 對話**（適合不確定從哪裡開始的人）：「我的工作是 ___，幫我拆解我的工作流程」
- **方法 C — 師父訪談萃取**（適合資深夥伴）：另一位夥伴擔任訪談者，用双云「師父知識萃取」框架

#### 流程拆解表模板

> 每一個工作流程填寫一份。一個人通常會有 3–8 個核心工作流程。

```
╔══════════════════════════════════════════════════════════════════╗
║                    流程拆解表 v1.0                                ║
╠══════════════════════════════════════════════════════════════════╣
║  填寫人：____________    日期：____________                      ║
║  流程名稱：____________                                          ║
║                                                                  ║
║  ─── 基本資訊 ───                                                ║
║  • 這個流程是在做什麼？（一句話）                                  ║
║  • 誰會觸發這個流程？                                              ║
║  • 做完之後的產出是什麼？                                          ║
║  • 多久做一次？  □ 每天 □ 每週 □ 每月 □ 不定期                   ║
║  • 每次大約花多少時間？  ____ 分鐘                                 ║
║                                                                  ║
║  ─── 步驟拆解（每步回答 3 個問題）───                            ║
║  步驟 N：________________________________________                ║
║    做什麼：                                                       ║
║    怎麼判斷做得對不對：                                            ║
║    常見的坑 / 眉角：                                              ║
║  （依此類推，通常 3-10 步）                                       ║
║                                                                  ║
║  ─── 判斷邏輯 ───                                                ║
║  • 哪些地方需要「憑經驗判斷」？                                    ║
║  • 有沒有「如果 __ 就 __」的規則？                                ║
║                                                                  ║
║  ─── 資料來源 ───                                                ║
║  • 需要參考哪些資料？（客戶紀錄 / 產業報告 / 範本 ...）           ║
║  • 這些資料目前放在哪裡？                                          ║
║                                                                  ║
║  ─── AI 適配評估 ───                                             ║
║  • 哪些步驟 AI 可以完全接手？                                     ║
║  • 哪些 AI 只能輔助、最後人判斷？                                 ║
║  • 哪些一定要人做、AI 碰不了？                                    ║
╚══════════════════════════════════════════════════════════════════╝
```

> **Jacky 註解**：「大部分人卡在這一步，因為他們覺得自己的工作『很難解釋』。其實不是難解釋，是你太熟了，所以跳過了很多你自己都沒意識到的步驟。這就像開車——你不會覺得『看後視鏡』是一個步驟，因為你已經內化了。但教新手的時候，你一定會講。」

### Phase 2｜生成 Agent 說明書（AI 自動轉換）

把 Phase 1 的流程拆解表，**貼給 AI**，請 AI 轉成 Agent 說明書。AI 完成、人審核。

#### Agent 說明書 9 區塊（模板）

```
╔══════════════════════════════════════════════════════════════════╗
║                   Agent 說明書 v1.0                              ║
╠══════════════════════════════════════════════════════════════════╣
║  1. 角色定位        — Agent 名稱 / 你是誰 / 服務對象 / 核心價值   ║
║  2. 觸發條件        — 什麼時候啟動 / 什麼時候不是你的事            ║
║  3. 輸入需求        — 必要 / 可選 / 缺欄位時主動問什麼             ║
║  4. 執行流程        — Step N：動作 / 判斷規則 / 輸出               ║
║  5. 判斷邏輯與決策樹 — 如果 A 則 B；無法判斷則回報人類             ║
║  6. 輸出規格        — 格式 / 品質標準 / 範例                       ║
║  7. 知識庫 / Dataroom — 資料類型 / 來源 / 更新頻率 / 結構          ║
║  8. 邊界與禁止事項   — 絕不做 / 需要交回人類的情況                ║
║  9. 進化機制        — 紀錄什麼 / 何時改說明書 / KPI                ║
╚══════════════════════════════════════════════════════════════════╝
```

> **Jacky 註解**：「說明書的靈魂在第 5 區（判斷邏輯）和第 8 區（邊界）。大部分人寫的 Agent 說明書只寫了『你要幫我做什麼』，但沒寫『什麼情況下該怎麼判斷』和『什麼事情你不能做』。結果 Agent 要嘛亂做、要嘛不知道怎麼做。」

#### 銜接關鍵

AI 生成的說明書一定要**人工審核**。重點看：判斷邏輯有沒有寫對、邊界有沒有畫清楚、知識庫需求是否完整。

### Phase 3｜部署上線（人選工具，AI 吃說明書）

#### 三大平台部署對照

| 項目 | Claude Skill | Gemini Gem | OpenClaw |
|---|---|---|---|
| 說明書放哪裡 | `SKILL.md` 檔案 | Instructions 欄位 | Agent 設定頁 |
| Dataroom 怎麼餵 | 放在 Skill 資料夾內 | 上傳到 Knowledge | 連接資料來源 |
| 觸發方式 | 對話中自動觸發 | @Gem 呼叫 | 自動／手動觸發 |
| 適合情境 | 深度分析、長流程、文件產出 | 搜尋整合、Gmail / Calendar | 多平台串接、自動化排程 |
| 團隊協作 | 分享 Skill 資料夾 | 分享 Gem 連結 | 分享 Agent 設定 |
| 進化方式 | 修改 `SKILL.md` | 修改 Instructions | 修改設定 + 重新訓練 |

#### 部署四步

1. **選平台** — 根據使用場景。不確定就先用最常開的那一個
2. **貼說明書** — Markdown 格式可能需要微調（Gem 去掉表格框線、OpenClaw 對齊結構化欄位）
3. **餵 Dataroom** — 把第 7 區列出的資料上傳到知識庫
4. **測試對話** — 用 3 種場景測：
   - 🟢 標準情境（流程跑得通）
   - 🟡 邊界情境（會不會越界）
   - 🔴 錯誤情境（怎麼處理錯資訊）

#### 部署檢查表

```
╔══════════════════════════════════════════════════════════════════╗
║  Agent 名稱：________________  平台：________________            ║
║  部署日期：________________    部署人：________________          ║
║                                                                  ║
║  □ 說明書已貼入平台指令欄位                                      ║
║  □ Dataroom 資料已上傳（共 ___ 份文件）                          ║
║  □ 觸發條件已設定                                                ║
║  □ 標準情境測試通過                                              ║
║  □ 邊界情境測試通過（Agent 正確回報「這不是我的事」）             ║
║  □ 錯誤情境測試通過（Agent 正確提醒異常）                        ║
║  □ 輸出格式符合說明書規格                                        ║
║  □ 已記錄第一版的已知限制與待改善項目                             ║
╚══════════════════════════════════════════════════════════════════╝
```

> **Jacky 註解**：「測試的時候你要刻意去刁難你的 Agent。不要只問它『幫我做個報告』然後覺得它好棒棒。要問『如果客戶臨時改需求怎麼辦？』、『如果資料不完整你怎麼處理？』——這些才是真實工作會遇到的狀況。測試不到位，上線就出包。」

### Phase 4｜迭代進化（持續餵養）

每次使用 Agent 後花 2 分鐘記錄。

#### 進化日誌模板

```
╔══════════════════════════════════════════════════════════════════╗
║  Agent 名稱：________________  日期：________________            ║
║                                                                  ║
║  • 今天用它做了什麼？                                              ║
║  • 它做得好的地方：                                                ║
║  • 它做得不好的地方 / 需要人工修正的地方：                         ║
║  • 是否需要修改說明書？  □ 不需要 □ 需要：                        ║
║  • 是否需要補充 Dataroom？  □ 不需要 □ 需要：                     ║
║  • 滿意度：□ 完全接手 □ 大致可用 □ 需大幅修改 □ 不堪用           ║
╚══════════════════════════════════════════════════════════════════╝
```

> **Jacky 註解**：「進化日誌是整個系統的飛輪。第一版的 Agent 一定只有 60 分，但每次用、每次修正，它就多 5 分。用 10 次就 80 分了。但如果不記錄、不回寫，它永遠停在 60 分。我自己的 Jacky Agent 已經迭代超過 30 次了。這不是因為 AI 變強了，是因為我持續把我的經驗餵給它。」

## Part J — 常見問題

**Q：我不知道我的工作哪些可以變成 Agent？**
A：問自己「這件事我每週做超過 2 次嗎？」如果是，就值得變成 Agent。先從最重複、最花時間的工作開始。

**Q：我的工作很難標準化，每次都不一樣怎麼辦？**
A：「每次都不一樣」通常是「判斷邏輯複雜」，不是「無法拆解」。把不一樣的部分整理成判斷規則（如果 A 就 B），剩下的標準步驟讓 Agent 做。**你只做判斷，Agent 做執行。**

**Q：三個平台我該選哪一個？**
A：選你現在最常開的那一個。不確定就選 Claude，它的指令理解能力目前最強。

**Q：Agent 做出來的品質不夠好怎麼辦？**
A：90% 的情況是說明書寫得不夠細。回去看判斷邏輯和輸出規格有沒有寫清楚。剩下 10% 是 Dataroom 不夠。**先改說明書，再補 Dataroom。**

**Q：同一個流程可以同時在多個平台建 Agent 嗎？**
A：可以，但沒必要。**說明書才是核心，工具只是載體。**

**Q：這跟之前建的 Agent 有什麼不同？**
A：之前建的 Agent 是「工具」，各自獨立。現在我們要把它們「串」起來，讓腦指揮手，讓老闆只管方向。

**Q：每個客戶都需要建一套嗎？**
A：對。每個客戶有自己的 4 Agent 委員會和 Google Form 資料庫。但 6 個手 Skill 是共用的，不需要重複建。

## Part K — Jacky 的最後一句話

> 「不要想一步到位。先拆一個流程、建一個 Agent、用一週、改一版。等你第一個 Agent 跑起來了，你自然會想做第二個、第三個。一年後你回頭看，你會發現你已經有一整個 Agent 團隊在幫你工作了。而這一切，都從填第一張流程拆解表開始。」
