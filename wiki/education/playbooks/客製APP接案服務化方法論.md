# 客製 APP 接案服務化方法論

> 把「客戶找上門問做 APP」這件零散接案的事，**包裝成可複製、可規模化的服務套餐**。第二次接類似案子，內部團隊照本流程跑，時間能砍掉 60%。
>
> **本教材定位**：商業流程 + 服務整合面（顧問流程怎麼跑、跟既有服務怎麼接）。
> **配對教材**：[APP 原型快速產出 SOP](APP原型快速產出SOP.md)（技術執行面）
> **來源**：J董 Labor APP 案 2026-05-19 從接觸到客戶簽認 MVP 一日內完成的實戰，抽象成可複製方法論。

---

## 一句話定位

**用「1 週可點原型」換「客戶決策 + 信任 + 簽約」**——技術只做到 HTML 模擬（[詳 SOP](APP原型快速產出SOP.md)），不寫真程式，但客戶能完整感受 80% 的功能。

> 双云行銷接案常碰到：客戶問「能不能也做 APP」。以前回答「可以但要先談 spec」，客戶往往就斷線。
> 現在的答法：「我們 1 週給你可點原型 + 14 張系統架構 + 交付清單，看完你再決定要不要簽。」

---

## 適用情境

| 對 | 不對 |
|---|---|
| 客戶手上有需求但沒 spec | 客戶已有完整需求書 + 開發團隊 |
| 客戶不一定懂技術 | 客戶 CTO 親自談規格 |
| 預算 30 萬 - 150 萬範圍 | 小於 30 萬（不划算）/ 大於 150 萬（要走正式投標） |
| 1-2 週內要從「想做」變「決定做」 | 客戶需要 3 個月以上的內部評估 |

## 7 階段顧問流程

### 階段 1 — 接觸與規格消化（1-2 天）

**動作**：
- 收到客戶任何形式規格（紙本拍照、txt、會議錄音）→ 進駐專案資料夾
- 不立刻動手寫程式，先做「規格盤點 + 隱含需求挖掘」
- 產出：給 J董/PM 的「現況盤點 briefing」+ 5 個必拍板選項

**為什麼**：80% 的客戶規格有 missing pieces，先挖出來再動工比 MVP 階段才發現便宜 10 倍。

**對應技術 SOP**：[階段 1 — 進駐與規格消化](APP原型快速產出SOP.md#階段-1--進駐與規格消化30-分鐘)

---

### 階段 2 — 三路市場掃描（半天 wall clock）

**動作**：派 3 個 AI agent 平行跑
- A：產業現況 + 法規 + 政府平台
- B：本土競品 + 功能對照 + 定價錨點
- C：GitHub 開源 + 技術選型建議

**目的**：證明三件事給客戶與 J董看
1. **市場有空白**（競品都沒做的功能組合）
2. **技術可行**（80% 站巨人肩膀 + 20% 自寫）
3. **客戶痛點真實**（法規 / 統計 / 媒體報導三方佐證）

**產出**：市場調查報告 + 高層 briefing（三色 callout：🟢時機 🟡警報 🔴風險）

---

### 階段 3 — 導覽頁 + 會議回填表單（半天）

**動作**：一個 HTML 解決三件事——
- 給客戶看的提案重點
- 給內部用的決策清單
- 給 J董會議用的回填表單（15 題決策節點）

**會議表單覆蓋的決策節點類別**：
| 類別 | 題目範例 |
|---|---|
| 必拍板（紅）| 商模、工種、預算、時程、雲端歸檔、技術路線 |
| 重要（黃）| MVP 功能切分、使用者角色、資料結構、業務員、Excel 整合 |
| 加分（綠）| 法規痛點驗證、政府 API、LINE 整合、資安門檻、競品認知 |

**設計重點**：推薦選項加綠底「推薦」標籤，方便 J董會議引導；不要做兩個檔，整合進同一檔。

---

### 階段 4 — 客戶會議與結論回填（半天會議 + 半天整理）

**會議 SOP**：
1. J董帶 `index.html` 入會場
2. 用導覽頁的「市場時機 + 競品稀缺」開場 5 分鐘
3. 逐題勾選 15 個決策節點（推薦選項先導引）
4. 自由筆記欄記客戶金句（之後做 pitch deck 用）
5. 會議結束按「產生會議結論」匯出 Markdown，貼給技術端

**會後 30 分鐘整理**：
- 存成 `會議結論_YYYY-MM-DD.md`
- 標**衝突警報**（預算 / 時程 / 範圍三角通常會打架）
- 列「8 項待客戶補充資訊」清單
- 列「客戶答應提供的資料」清單

**關鍵原則**：
- **預算/合約金額由 J董處理**——AI 出範圍 + 工時 + 風險，不出金額
- 客戶在表單裡決策比口頭討論積極 3 倍
- 衝突警報內部知道就好，**不在客戶面前打臉**

---

### 階段 5 — MVP 互動原型（1-2 天）

**動作**：用 HTML 模擬手機 APP，**不要真做 RN**。

**為什麼省 90% 時間還能感動客戶**：
- 真做 RN：環境設定 → 上架簽章 → 後端 API → DB schema 至少 2 週起跳
- HTML 模擬：手機框 + 假資料 + localStorage，1 天可點按 5 個畫面

**MVP 必備畫面**（依客戶業別調整）：
- 登入（不需密碼，直接進入）
- 主功能列表（含 3-5 個假樣本）
- 詳情頁 + 分類抽屜
- **拍照 OCR 流程**（最魔法的功能，5-6 步動畫）
- 主動提醒清單（紅黃綠燈分級）

**接客戶真實案例**（強烈建議）：
- 拿到客戶 5-10 份真實 PDF，放 `真實案例/` 資料夾
- MVP 中可點開原始 PDF（target=_blank）
- 客戶看到「自家資料在我們系統裡跑」會心理上**已當成自家產品**

**業務規則 chip 標籤**（重要！）：
- 抽象的「文件分類規則」客戶聽不懂
- 用彩色 chip 標籤（如 Labor 案的「雇主/受照顧者/移工」三色）
- **客戶看到 chip 才會校正規則**——這是 MVP 階段最大價值

**對應技術 SOP**：[階段 5 — MVP 互動原型](APP原型快速產出SOP.md#階段-5--mvp-互動原型1-2-天)

---

### 階段 6 — 教育訓練 + 客戶交付清單（半天）

**動作**：寫 `training.html` 14 張投影片，5 個 Part。

**最關鍵的設計**：**Part 4 客戶交付清單可勾選 + 一鍵匯出**
- 17 項分 A/B/C/D 四類（案件資料 / 員工權限 / IT 環境 / 法規評鑑）
- 客戶在 HTML 上直接勾選 → 按「匯出」→ Markdown 自動到剪貼簿
- 客戶貼 LINE/Email 回傳即可
- 比口頭交辦或 email 清單**回收率高 5 倍**

**附加產出**：客戶交付包資料夾
- 簡化版入口（不含內部用會議表單）
- 複製 mvp / pitch / training / 真實案例
- 純文字使用說明（IT 阿姨阿伯看得懂）

---

### 階段 7 — 客戶簽認 → 進入正式開發

**簽認觸發**：客戶簽認 MVP 原型 = 規格凍結 = 開發合約可簽
- 後續路線：技術 PoC → v1 → v1.5 → v2 → SaaS 化（兩階段商模）
- auto memory 寫進專案記憶，下次跨對話接續

## 7 大關鍵設計決策（為什麼這樣做）

| 決策 | 為什麼 | 採坑 |
|---|---|---|
| **用 HTML 模擬 APP 而非真做 RN** | 省 90% 時間，客戶感受 80% | 客戶第一次看到會問「這就是 APP 嗎」要先預告「示意原型」|
| **三路平行 agent 而非單線研究** | 客戶問「為什麼是現在/為什麼是你」要市場+競品+開源三維交叉 | 每個 agent prompt 要自成一體 |
| **會議回填表單而非筆記** | 客戶在表單裡決策積極 3 倍 | 推薦選項要標清楚，避免客戶亂選 |
| **預算讓 J董處理而非自動估** | 接案議價空間是商業判斷，AI 不替老闆決定 | 內部清楚講「金額由 J董寫」，避免 AI 自作主張 |
| **業務規則用彩色 chip 標籤露出** | 抽象規則客戶聽不懂，視覺一秒懂 | 規則一改 schema 就要升 localStorage key |
| **真實案例 PDF 接到 MVP** | 客戶看到自家資料會心理上當成自家產品 | 隱私邊界要交代「上線前去識別化」|
| **客戶交付包打包資料夾** | 客戶要拿著走內部討論 | 客戶版入口要簡化（不含內部用表單）|

## 服務化定價邏輯（給 J董參考的範圍框架，不含金額）

> **預算金額由 J董決定**，本表只提供「客戶投資 vs 我方產出」的對應關係，便於 J董開價時有依據。

| 服務套餐 | 時間 | 產出 | 客戶決策狀態 |
|---|---|---|---|
| **顧問諮詢套餐** | 2 週 | 市場調查報告 + MVP 原型 + 系統架構教材 + 交付清單 | 客戶決定是否動工 |
| **MVP 開發套餐** | + 3 個月 | 顧問套餐 + 真正 RN APP（v1） | 客戶內部試用 |
| **完整 v1.5 套餐** | + 1.5 個月 | + 60 天到期提醒 + 離線 + 多工種 | 自家用版本完整 |
| **v2 + SaaS 化** | + 2 個月 | + AI OCR + 2FA + 多租戶改造 | 開始賣同業 |

**兩階段商模**（適用於 B2B 工具）：
1. 自家用版本跑通（v1-v1.5）
2. 抽 SaaS 賣同業（v2 + 多租戶）

## 服務組合 cross-sell 範例（Labor 案推演）

> 假設 Labor 客戶順利簽約進入開發階段，後續可以延伸：

```
Labor APP（人力仲介數位管理）
  ├──→ 双云行銷品牌包裝（仲介公司形象再造）
  ├──→ AGENTS L1/L2 認證（仲介公司內部建 AI 行銷部）
  ├──→ TBSA 五張表單顧問（仲介公司年度戰略規劃）
  ├──→ AI 諮詢服務（仲介公司其他工具評估）
  └──→ SaaS 化後分潤（賣同業給 1,400 家中小仲介）
```

一個客戶 LTV（生命週期價值）能從「APP 開發 60 萬」放大到「持續服務 200+ 萬/年」。

## 異常情況分支

| 情況 | 處理 |
|---|---|
| 客戶當下決定不下來 | 表單先存草稿（localStorage），會後再決定 |
| 客戶在 mvp 階段提出全新需求 | 開新 feature 分支處理，不要污染原 main |
| 客戶預算遠低於可行範圍 | 提出方案 A/B/C 三選一，由 J董談判（我們不主動砍範圍） |
| 客戶簽認後又反悔 | 把已拍板事項寫進 auto memory，下次討論時可引述 |
| 客戶說「視覺先不用考慮」 | 全程不做精美設計，留 token 給其他工作 |
| 客戶業務領域我方不熟 | 三路 agent 階段加重「法規 + 業界術語」研究 |
| 客戶要看詳細報價但 J董還沒定 | 客氣回「J董會在 X 月 X 日跟您面對面說明」 |

## 快速回顧

把「客製 APP 接案」從**零散工作**變成**可複製產品線**的關鍵 4 件事：

1. **規格未明前不寫程式**——三路平行 agent 把市場 + 競品 + 開源證據抓齊
2. **HTML 模擬換時間**——MVP 原型不寫 RN，用最低成本換最高客戶決策
3. **業務規則用視覺 chip 露出**——客戶在 MVP 階段才會校正規則，這是最大價值
4. **預算讓老闆處理**——AI 出範圍 + 時程 + 風險，金額是商業判斷不是技術判斷

第二場、第三場客戶來時，內部團隊照本流程跑，時間能砍 60%、客戶簽認率能拉到 80%+。
