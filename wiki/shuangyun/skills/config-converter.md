# shuangyun-config-converter｜設定檔轉換（流程 → Agent）

> **層級**：Hand（5 助教 Agent 之一）
> **狀態**：✅ 已建（v1.0 / 2026-03-12）
> 來源：`06_Skills/shuangyun-config-converter/`

---

## 一句話定位

> **Agent 設定檔產生器**：讀補完後的拆解表，自動轉換為 9 大區塊通用設定檔 + 直接可貼進 Gem / GPTs / Skill / NotebookLM 的角色指令。

> 手動寫 20-30 分鐘 → 此 Skill 2 分鐘完成，格式一致、不漏項。

## 觸發條件

- 「轉設定檔」「拆解表轉 Agent」「產生設定檔」「設定檔轉換」
- 「9 大區塊」「Agent 設定檔」
- 學員貼上拆解表希望轉 Agent 格式
- 課堂進入「流程轉 Agent」實作環節
- 「角色指令」「Instructions」「部署格式」

## 執行流程（4 步）

### Step 1：解析拆解表

提取：流程名稱、一句話描述、觸發情境、步驟清單、判斷規則、資料來源、產出物、眉角。

**缺漏處理**

| 情況 | 處理 |
|---|---|
| 判斷邏輯完全空白 | 提醒「建議先用 [completion-coach](completion-coach.md) 補完再轉」，但不阻斷 |
| 其他缺漏 | 標記為「待補」，繼續轉換 |

### Step 2：映射到 9 大區塊

| 拆解表區塊 | → | 設定檔區塊 | 轉換規則 |
|---|---|---|---|
| 流程名稱 + 一句話描述 | → | 1. 角色定位 | 改寫為第二人稱：「你是__，專門負責__」 |
| 觸發情境 | → | 2. 觸發條件 | 列出關鍵字和場景 |
| 步驟的輸入 | → | 3. 輸入需求 | 區分必要和可選 |
| 步驟拆解 | → | 4. 執行流程 | 保持原始順序，每步加「動作 / 判斷 / 輸出」 |
| 判斷邏輯 + 眉角 | → | 5. 判斷邏輯 | 整理為「如果__就__」格式 |
| 產出物 | → | 6. 輸出規格 | 加上格式要求和品質標準 |
| 資料來源 | → | 7. 知識庫需求 | 整理為 Dataroom 資料夾結構建議 |
| AI 適配中「不能做」的 | → | 8. 邊界禁止 | 改寫為「你絕對不能__」 |
| （自動生成） | → | 9. 進化機制 | 根據流程特性建議追蹤項目 |

### Step 3：生成角色指令核心段

另外產出 200-400 字的精簡版「角色指令」，直接貼到 Gem / GPTs / Skill 的 Instructions 欄位。

包含：你是誰 + 任務 + 限制 + 輸入輸出 + 關鍵判斷規則。

### Step 4：輸出完整設定檔

Markdown 格式輸出 9 大區塊設定檔，末尾附：
- **平台部署對照表**（Gem / GPTs / Claude Skill / NotebookLM）
- **「待補」清單**（拆解表中沒寫到的部分）

## 判斷邏輯

**角色指令語氣**

| 流程類型 | 語氣 |
|---|---|
| 面對客戶（月報、提案） | 專業正式 |
| 內部作業（排程、整理） | 簡潔直接 |

**拆分判斷**

| 觸發 | 動作 |
|---|---|
| 步驟超過 8 步 | 建議拆成 2 個 Agent，說明切分點 |
| 一個步驟有 2+ 判斷點 | 標記「建議將此步驟拆成獨立 Agent」 |

**缺漏標記**

- 判斷邏輯空白 → 第 5 區寫「⚠️ 待補——目前 Agent 遇到判斷情況會直接詢問使用者」
- 資料來源空白 → 第 7 區寫「知識庫待建——首次使用前需盤點上傳」

## 輸出規格

- 角色指令必須**第二人稱**（「你是」不是「這個 Agent 是」）
- 每個步驟有明確動作描述
- 判斷規則必須可執行（「如果__就__」而不是「看情況」）
- 設定檔可直接複製貼上到任何平台使用

## 邊界

- 不能「發明」拆解表中沒有的步驟或規則
- 不能修改學員描述的流程順序（除非明顯有邏輯矛盾）
- 不能省略拆解表中的任何步驟
- 步驟之間有邏輯矛盾 → 交回人類
- Agent 功能明顯超出原始流程範圍 → 交回人類

**追蹤 KPI**：設定檔首次部署的「直接可用率」（學員拿到後不需大幅修改就能用的比例）
