# 上線前容量壓測體檢法（AGENTS【修】的進階版）

> **一句話**：AGENTS 的【修】(E 評測修正) 原本測「**內容對不對**」（三色情境殘酷測試）；本法把同一套「殘酷測試精神」延伸到「**效能/容量跑不跑得動**」——上線/demo 前強制實測算數字，給有憑有據的 GO／NO-GO，杜絕「操作後才發現跑不動」。
> **配對 skill**：`/壓測`（capacity-check）。
> **配對模式**：[高併發活動撐流量設計法](高併發活動撐流量設計法.md)＝「**建**」（為流量而建）；本篇＝「**修**」（上線前驗證）。
> **來源（觸發契機）**：2026-06-26 agents.tbsa.tw 6/26 說明會——AI 只驗了「護欄設定對」就說「30–50 人沒問題」，沒實測 → Jacky 當眾 demo 時記憶體不足轉圈圈跑不出來。本法是那次事故的反推產物。

---

## 一、為什麼是 AGENTS【修】的進階版

AGENTS 六字訣「拆→建→**修**→串→管→交」中，**【修】= E 評測修正**，核心是**殘酷測試**：用 🟢標準／🟡邊緣／🔴錯誤 三色情境去打 Agent，逼出它在內容上的破綻，產出測試報告與 v2。

但 Agent / 服務上線會死，不只死在「**內容錯**」，更常死在「**跑不動**」：

| 品管軌 | 問哪個問題 | 原本【修】 | 本法（進階【修】）|
|---|---|---|---|
| **內容軌** | 答得對不對？ | ✅ 三色情境殘酷測試 | （沿用）|
| **效能軌** | 撐不撐得住、夠不夠快？ | ❌ 常被略過 | ⭐ 四層容量壓測 |

> 這正是 wiki 既有「**雙軌品管**」（事前可追溯性 ＋ 事後殘酷測試，源自 73% 幻覺事故）的延伸——把「殘酷測試」這條軌，從**內容正確性**再長出一條**效能/容量**分支。

**核心心法**：「**護欄設定對** ≠ **體驗順**」。設定再漂亮，沒實測過一次最重操作的秒數、沒看過整台機器的記憶體餘量，結論一律是「**未驗證**」。

---

## 二、鐵律（先記住，否則又會錯）

1. **不准說「應該 OK」，只准說數字。** 沒有秒數＋餘量數字之前，結論是「未驗證」。
2. **容器用量 ≠ 整台 VM 用量。** 永遠看整台伺服器 RAM 餘量，不是單一容器。（事故主因：只看容器 66MB，沒看整台 76%。）
3. **Build 在伺服器上跑、瞬間吃 0.5–1GB。** 一台已 75% 的機器，一次部署就 OOM 殺容器→整站掛。
4. **外部額度/自動儲值要實查、且以「後台」為準。** API 推算（`credits − usage`）會被帳務模型/快取扭曲，只能當參考並標明，**不可當事實報出**。
5. **一定要親手跑一次「最重的操作」看碼表**（不是看設定）。LLM 生成可能 80 秒。
6. **壓測不准灌正式站**（≤5 併發小探針可以；大量壓測去 staging／本機）。
7. **活動當天不部署**（任何 build/重啟＝一次停機窗口）。

---

## 三、四層瓶頸模型（教學主體）

服務會炸不是單一原因。按四層查，**每層都要拿到數字**，缺一不可：

### L1 — 伺服器 VM 餘量
- 查：整台 VM 的 RAM/CPU/Disk 餘量、同台**共用租客**數（多服務搶同一台＝互相吃 RAM）。
- 紅旗：RAM >75% 或剩 <1.5GB；多個服務擠一台 2GB。
- 工具（Zeabur）：`server get`（看 `Status.UsedMemory/TotalMemory`，**不是容器 metric**）；`project get` 看在哪台；逐專案 `service list` 數租客。

### L2 — 建置（Build）會不會把它榨爆
- 查：部署時的 RAM 尖峰（Node/Vite/tsc build ~0.5–1GB）vs 伺服器剩餘 RAM。
- 算式：`build 尖峰(≈1GB) ≤ 剩餘 RAM`？不夠就會 OOM。
- 解：升級規格／搬到有餘量的機器／改用預建 image（不在這台 build）。

### L3 — 應用/執行期延遲
- 查：**最重操作**（生成/匯出/查詢）親手跑 3 次取中位數＋ ≤5 併發小探針。
- 紅旗：互動頁 >3s；重操作（demo 用）>30s；小併發下出現 5xx/逾時。
- 工具：`capacity-probe.sh`（量 `time_starttransfer`/`time_total`/中位數，CONC≤5）。

### L4 — 上游/外部限制
- 查：LLM/API 的**餘額、每 key 速率/日上限、自動儲值是否真會跳**。
- 算：`可跑次數 = 可用餘額 ÷ 單次成本`；要 `≥ 預估用量 × 1.5`。
- 紅線：餘額**以使用者後台為準**；API 推算值標「約、需核對」，不可斷言「你剩 $X」。

## 四、執行流程

0. **問清場景**：尖峰同時在線？單日總次數？最重操作是什麼？何時用（離峰/活動當下）？沒數字就用保守假設並寫明。
1. **L1** 伺服器餘量 → 拿數字。
2. **L2** build 尖峰 vs 餘量 → 算夠不夠。
3. **L3** 跑最重操作計時 ＋ ≤5 併發探針 → 拿秒數。
4. **L4** 實查上游額度/速率 → 算可跑次數。
5. **降級驗證**：額度爆/併發滿/上游錯誤時，使用者看到的是**友善引導**還是醜 5xx/空白轉圈？實際打到那情境確認。
6. **出報告**：GO／慎用／NO-GO ＋ 最先炸的瓶頸層 ＋ 具體解法，全部附「現在這秒」的數字與時間戳。

### 報告格式
```
# 容量體檢：<服務> @ <時間>
場景假設：尖峰 <N> 併發 / 單日 <M> 次 / 最重操作 <X>
L1 伺服器：RAM <used>/<total>MB（剩 <free>）| 租客 <k>      → 🟢/🟡/🔴
L2 建置  ：build ~<?>GB vs 剩 <free>                         → 🟢/🟡/🔴
L3 延遲  ：首頁 <t>s | 最重操作中位 <t>s | <5併發最慢 <t>s   → 🟢/🟡/🔴
L4 上游  ：餘額 ~$<b>（以後台為準）÷ 單次 ~$<c> = 約 <ops> 次 → 🟢/🟡/🔴
判定：✅GO / 🟡慎用(條件…) / 🔴NO-GO ｜瓶頸：<層> ｜要做：<具體>
```

## 五、案例：agents.tbsa.tw（事故 → 體檢）

| 層 | 事故當時（2GB） | 升級後（8G）|
|---|---|---|
| L1 | RAM 76%、剩 470MB、7 租客 🔴 | 30%、剩 5.3GB 🟢 |
| L2 | build 1GB > 470MB → **OOM** 🔴 | 1GB << 5.3GB 🟢 |
| L3 | 首頁 0.3s🟢；**生成 83 秒** 🔴 | 同（待降 max_tokens/換 provider）|
| L4 | 餘額見底、自動儲值沒跳 🔴 | （以後台為準）|

**結論**：基礎設施紅旗用「升級伺服器規格」解掉（2GB→8G）；L3/L4 是另外的待辦。**每一項都是「跑一下就會發現」的——當初沒跑，才會當眾出事。**

---

## 六、紅線與邊界
- ❌ 不對正式站灌高併發；不在活動當天/demo 中部署；不憑設定下結論。
- ✅ 探針用真實但少量請求，測完清掉測試資料；報告附「現在這秒」數字。
- 非 Zeabur：L1/L2 改用該平台資源面板／`free -m`／`docker stats`；L3/L4 通用。
- 沒 staging：高併發就不做，只做 ≤5 探針並老實標「未做高併發壓測」。
