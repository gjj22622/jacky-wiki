# AI 作業系統觀

> **核心信念**：AI 不是一個要「學」的工具，而是一套要「進駐」的**作業系統**。
> 以前我用 Windows、別人用 Mac，現在我用的是 **AI 系統**——透過四種介面（web／手機／desktop／終端機）操作同一套環境。
> 大家還在焦慮「要不要學 AI」時，真正發生的是：**工作的作業系統正在換代**。換 OS 不是學 App。

這頁是 [AI 工具觀](AI工具觀.md) 的上位。工具觀講「**選哪些工具**」；作業系統觀講「**把工具組成一套個人 OS、定義每日入口與工作流**」。

---

## 為什麼是「作業系統」而不是「工具」

工具是 stateless 的一次性呼叫；作業系統是持續運行、有記憶、會調度、能決策的環境。這個區分不是比喻修辭，業界已有對應架構（見下表）：

| 面向 | 工具（Tool） | 作業系統（OS） |
|---|---|---|
| 生命週期 | 一次呼叫即結束 | 長期運行、持續推理 |
| 記憶 | 無狀態 | context（工作記憶）＋ Memory／Wiki（長期記憶） |
| 決策 | 你下指令它做 | 自主規劃、調度多步驟 |
| 角色 | 邊邊角角的功能 | 編排整個工作流（如 OS 管理行程） |

**換個視角看你現在怎麼用 AI**：你不是在「用 ChatGPT / Claude / Codex 這幾個工具」，你是在**為自己組一套作業系統**——這些工具是這套 OS 的元件。看清楚這件事，就會知道自己的工作流該調整成什麼樣。

---

## 四種介面 = 同一套系統的四個呈現

| 介面 | 定位 | 我的用法 |
|---|---|---|
| **終端機**（最常用）| OS 的 shell——可程式化、可自動化、可串接 | Claude Code / Codex CLI，主工作場 |
| **desktop** | 連本機工具、長任務、connector | Cowork、長期工作流 |
| **web** | 策略討論、起稿、artifact | 腦力激盪、初稿 |
| **手機** | 隨身入口、雲端 agent 續跑 | 移動中下指令、追任務 |

**關鍵**：這不是四個不同的工具，是**同一套 OS 的四種 UI**。業界已有實證——Cursor CLI（2026-01）做到同一個 agent harness、相同記憶、相同 MCP 連接，在 IDE／終端／Web／Mobile 之間無縫切換。我的四介面觀正是這種架構的日常實踐。

## 兩大進入障礙（一般人卡在哪）

進入這個作業環境的門檻不在「會不會用某個工具」，而在兩件事：

### 障礙 1：環境的建立
- 安裝與接線：終端機、Node、Docker、CLI 工具、MCP connector、帳號權限。
- 業界數據佐證：46% 企業把「與現有系統整合」列為 AI 導入首要挑戰；「context engineering」被評為 agent 工程師的第一號工作——這就是新的環境設定複雜度。

### 障礙 2：使用的流程與步驟
- 不是會開工具，而是知道「每天從哪個入口進、用什麼流程把事做完、怎麼驗收」。
- 沒有流程，AI 只是好玩一兩次的對話；有了流程，才會套回原工作流並**大升級**。

> 這兩件事一旦建立好、套進原本的工作流，工作流程就會跳級。糾結「學不學 AI」是錯的問題；對的問題是「**我的作業系統建好了沒、流程定義了沒**」。

## 作業系統的架構層（把直覺升級到架構）

借業界已驗證的對應（Karpathy「Software 3.0 / LLM OS」、AIOS、MCP），把「感覺像 OS」說成「架構就是 OS」：

| OS 概念 | AI 對應 | 在我的體系 |
|---|---|---|
| CPU | 模型 weights（固定計算引擎） | Opus / GPT-5.4 等 |
| RAM | context window（揮發工作記憶） | 當前對話 context |
| 程式語言 | 自然語言 prompt | 「英文是最熱的程式語言」 |
| 行程 Process | Agent（有生命週期、記憶、優先級） | Subagent / Skill 執行單位 |
| 系統呼叫 | MCP / 工具呼叫 | Drive / Gmail / Calendar MCP |
| 檔案系統＋長期記憶 | Memory / Wiki / git repo | auto-memory、jacky-wiki |

對應 [[AGENTS]] 的「腦＋手」架構：腦＝文件＋AI 判斷（決策層／kernel），手＝Skill 執行（行程／工具層）。

## 為什麼用「作業系統邏輯」帶新人更好

> 新人進双云，不是學一堆 AI 工具，而是**進駐一套新的作業系統**——就像當年第一次學 Windows。

- **逐個教工具是錯的**：工具難教、又變化太快，今天教的介面三個月後就改版。
- **OS 邏輯可遷移**：教「為自己組一套作業系統＋定義每日入口與工作流」，這套邏輯不隨工具汰換而失效。
- **帶人更好帶**：新人有了 OS 的心智模型，遇到任何新工具都知道它在系統裡的位置（是 shell？是 connector？是長期記憶？），不必每次從零。

這也是双云 AI School 的任務之一：成為這個**作業平台的入口**——先讓人換上 OS 視角，再談工具。

## 三個常見誤區

### 誤區 1：把 AI 當「多開幾個對話的工具」
症狀：在 web 開 50 個對話，沒持久化、沒 git、每次重講 context。
解法：對話只做腦力激盪；落地一律回終端機／檔案系統（OS 的 shell 與檔案系統）。

### 誤區 2：環境沒建好就急著上工
症狀：跳過環境建立，直接想要產出，結果卡在接線與權限。
解法：先把「環境建立」當第一道工程——終端、CLI、MCP、權限、記憶就位，再談流程。

### 誤區 3：逐個追工具、忘了系統
症狀：每出新模型/新工具就跳，沒有累積成一套可運行的 OS。
解法：用「這個元件補我系統的哪個缺口」來決定要不要納入（延續 [AI 工具觀](AI工具觀.md) 的加減工具規則）。

## 需要補強的面向（誠實的限制）

- **記憶分層**：context=RAM 只是工作記憶，還需長期記憶（持久商業事實）與兩者的切換策略；超出 context 上限時的「AI archaeology」問題。
- **權限與治理**：多 agent 存取敏感資料、人類監督有限——安全／合規／審計成為這套 OS 的核心問題，不是附加。
- **可靠性-ROI 缺口**：業界預估到 2027 年約 40% agentic 專案因可靠性與 ROI 不明而取消；demos 不等於 products，流程與驗收的工程紀律仍要做足。

## 業界佐證（寫教材／對外講述時引用）

- Karpathy「Software 3.0 / LLM as OS」：context=RAM、weights=CPU、prompt=程式語言。
- 微軟 Copilot：「Intent as Kernel, Language as Shell, AI as Runtime」——OS 層的自主編排，不只 UI 層。
- Cursor CLI（2026-01）：同一 agent、四介面（IDE／終端／Web／Mobile）同核心、無縫切換——四介面觀的實證。
- AIOS（Rutgers）：把 Agent 映射成傳統 OS 的行程（排程、context 切換、記憶、存取控制）。
- a16z / Goldman Sachs / 微軟：2026 是「代理 OS 時代」的臨界點，AI 從工具升級為環境/系統/代理。
