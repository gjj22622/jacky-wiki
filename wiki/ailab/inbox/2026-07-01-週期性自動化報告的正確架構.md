## 發生了什麼
要「每月 1 號自動產出營運盤點報告＋策略」。第一直覺是用 `/schedule` 開**雲端 routine**（cron cloud agent），但發現雲端 agent 只能 checkout git repo，**讀不到 gitignored 的客戶機密**（CRM 真相 `clients/**` 全被 gitignore）。改用**本機 systemd user timer** 才對——資料都在本機。

## 為什麼重要
這是「週期性 AI 自動化」選型的通用判準，之前沒明講：
- **雲端 routine 的死穴＝看不到本機/機密資料**。凡是報告依賴 gitignored 檔、本機 SQLite、local env、daemon 產物 → 一律本機排程，不上雲。
- 報告類自動化要拆 **90/9/1**：90% 確定性收數據（腳本讀 case.json/日誌/訂單，零 LLM、零幻覺）→ 9% LLM 只做「合成策略」這個分岔點（`claude -p` headless）→ 1% 對外動作（上傳/通知）而且報價等紅線仍停人。
- 附帶價值：確定性那層可加 `--data-only` 乾跑模式，隨時安全查數字、不觸發 LLM/commit。

## 怎麼做的
1. **確定性收集器**（Node 零依賴）：讀真相源算統計，組成 dataBlock 文字。
2. **LLM 合成**：`execFileSync("claude", ["-p", prompt, "--model", "claude-sonnet-4-6"])`，prompt 內灌入 dataBlock ＋上月報告＋角色原則，明令「數字只能用 dataBlock、禁杜撰金額案號」。失敗則 fallback 純數據版。
3. **對外**：git commit+push wiki → `gog drive upload` → Telegram 通知，每步 try/catch 降級不中斷。
4. **排程**：`~/.config/systemd/user/<name>.{timer,service}`，timer 用 `OnCalendar=*-*-01 01:00:00 UTC`（= 台灣 09:00）+ `Persistent=true`（漏跑會補），service 是 `Type=oneshot`。service 要顯式 `Environment=PATH=...hermes/node/bin:...` 否則 systemd 找不到 node/claude/gog。
5. `systemctl --user daemon-reload && enable --now <name>.timer`，用 `list-timers` 驗證 NEXT 時間。

## 對比與替代
- **雲端 routine（否決）**：讀不到機密、每次冷啟零上下文；適合純 public repo 的任務，不適合吃本機真相的報告。
- **crontab（可用但較弱）**：無 `Persistent` 補跑、無單元狀態查詢、log 要自己接；systemd user timer 在可觀測性上勝出，且與現有 daemon 家族一致（照抄 `jdong-journal.timer` 範式最省事）。
- **純 LLM 生報告（否決）**：會幻覺金額/案號。90/9/1 把數字鎖死在確定性層是關鍵。
