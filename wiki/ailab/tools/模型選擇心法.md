# 模型選擇心法

> **核心原則**：不是用最強，是用最對。
> **方法論**：根據任務性質分流到不同模型／工具。

---

## Claude 三檔選擇

### Opus 4.7 (1M context)
| 特徵 | 適用 |
|---|---|
| 最深、最會推理、最貴 | 方法論、架構、長文寫作 |
| 1M context（5min cache）| Wiki ingest、長文獻整合 |
| 不適合 | 重複性、批次任務（太貴）|

### Sonnet 4.6
| 特徵 | 適用 |
|---|---|
| 快、穩、CP 值高 | 一般 coding、日常任務 |
| 標準 context | 一般長度 |
| 不適合 | 跨檔結構大重構（漏改）、長 context（不夠）|

### Haiku 4.5
| 特徵 | 適用 |
|---|---|
| 快、便宜、深度有限 | Subagent 內呼叫、批次 grep |
| — | — |
| 不適合 | 主對話、深度任務 |

## 1M Context 的真正用法（不是濫用）

Opus 4.7 的 1M context 看起來很大，但**有 5 分鐘 cache TTL**：
- 5 分鐘內重用 → cache hit，便宜
- 超過 5 分鐘 → cache miss，全文重讀（貴 + 慢）

**正確用法**：
- ✅ 一次把大量素材（OneDrive 多檔）讀進來，5 分鐘內密集處理
- ✅ Wiki ingest：素材讀進來 → 連續寫多個頁面 → 同一 cache window
- ❌ 把 1M context 當「永久記憶」（不是，cache 會過期）
- ❌ 隨手讀大檔然後 1 小時後再用（cache miss）

## 何時切回網頁版（Claude Web / ChatGPT Web）

| 場景 | 為什麼用網頁 |
|---|---|
| 純 brainstorm（不需落檔）| IDE 過度結構化反而慢 |
| Computer 不在身邊 | 手機 / 平板 |
| 給別人看 demo | 不想讓人看本機檔案 |

> 但**所有要落檔的工作都回 Claude Code**——不要在網頁版做正事。

## 模型對比的紀錄方法

每次發現新差異，用 [實踐捕手協定 type=模型對比](../concepts/實踐捕手協定.md) 記。

範例事件累積後升格本頁規則：

| 累積事件 | 升格成 |
|---|---|
| 「Codex 跨檔比 Sonnet 穩」x 3 | 主對照表「跨檔結構重構 → Codex」|
| 「Opus 文案比 GPT-5.4 自然」x 2 | 主對照表「文案／創意 → Opus」|
| 「Haiku Subagent 內呼叫比 Sonnet 快 3x」x 2 | 主對照表「Subagent 內 → Haiku」|

---

## 模型版號更新時的處理

當 Claude / GPT 出新版（如 Opus 5、GPT-6），走 [學習方法](../concepts/學習方法.md) 三階段：

1. Phase 1（24h，介面同）：跑 3 個迥異任務看能力差異
2. Phase 2（1-2 週）：刻意分流，並行使用對比
3. Phase 3：更新本頁主對照表

> 不要看到新模型就直接全切——並行驗證後再決定。

## 限制與雷區

| 雷 | 解 |
|---|---|
| 一個模型包山包海 | 分流（本頁）|
| benchmark 領先就切 | benchmark ≠ 你的場景，自己驗證 |
| 1M context 當永久記憶 | 是 5 分鐘 cache，不是 memory |
| Opus 拿來跑 Subagent | 太貴，用 Haiku |
| Sonnet 拿來跨檔大重構 | 漏改，用 Codex |
