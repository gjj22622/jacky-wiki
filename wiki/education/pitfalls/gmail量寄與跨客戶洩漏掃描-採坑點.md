# 採坑點——Gmail 量寄與跨客戶洩漏掃描

> 做活動後的批量跟進信，一路踩到的雷。每筆統一欄位，照著避。

## 採坑 #1：「看板沒資料」≠「沒發生」
- **症狀**：演講後看 CRM 看板與站台 leads buffer，幾乎看不到名單，差點對外回報「演講零轉換」。
- **誤區假設**：看板查不到 = 名單沒進來 = 演講沒成果。
- **真實原因**：73 個 lead 全都寫進磁碟 `case.json`，但建案順序是「寫檔 → git commit → 投影 read model」，而 `clients/**` 退版控後 commit 報錯 → `ok:false` → **檔案已落地卻從沒投影**。看板/buffer 是衍生層，壞了。
- **解法**：磁碟才是真相來源；用 `rebuildReadModel()` 從磁碟全量重建。git-store 對被 gitignore 的路徑用 `check-ignore` 優雅跳過。
- **預防**：下任何「沒成果」結論前，先問「是真的沒有，還是我查的那層（read model/buffer/快照）壞了？」先驗觀測管道。
- **發生於**：2026-06-13 演講成果盤點。

## 採坑 #2：空頭承諾——不問就說「我幫你撈」
- **症狀**：診斷跟進信寫「我會用明察為您的產業撈一份情報」，但根本沒問對方是什麼產業、什麼關鍵字。
- **誤區假設**：先承諾，細節之後再說。
- **真實原因**：明察撈標案是 `標案名稱 LIKE '%關鍵字%'`，沒有具體產品關鍵字根本撈不到；硬做就得編 → 踩反幻覺紅線。
- **解法**：把承諾改成「先問再做」——當場補建 `/diagnosis` 結構化問卷頁，收「產品關鍵字／是否做政府生意／想看哪種情報」等明察真正需要的輸入，再進產線。
- **預防**：寫任何「我幫你做 X」前先問「我手上有沒有做 X 的必要輸入與資料？」沒有就先收集。
- **發生於**：2026-06-15 審到診斷批。

## 採坑 #3：用 import「測試」有副作用的腳本 = 誤觸真寄
- **症狀**：想驗證寄信腳本的標頭剝除，跑了 `node -e 'import("./send-followup.mjs")'`，結果直接啟動真寄迴圈（每 2 分一封）。
- **誤區假設**：import 只是載入，不會執行。
- **真實原因**：ESM `import` 會執行模組**頂層程式碼**；該腳本頂層就是寄送主流程。
- **解法**：立刻 `pkill`；幸好當下洩漏掃描把全部擋下、**0 封真的寄出**。
- **預防**：驗證有副作用的腳本一律用 `--dry-run` 旗標，**絕不**用 import/require 去「測」。
- **發生於**：2026-06-15 建寄送器時。

## 採坑 #4：跨客戶洩漏掃描誤擋「自己人/場地/公開產品名」
- **症狀**：每封跟進信都被 leak scan 擋下，理由是「含其他客戶『鐘基啟』『彰師大』『Google』」。
- **誤區假設**：掃描器報的都是真外洩。
- **真實原因**：署名本人（鐘基啟，且某自家 case 的 name 欄就是它）、演講場地（彰師大，某 lead 的 name 欄）、信裡「Google Meet」撞到名叫 Google 的潛客——都是公開/自家字，不是他人機密。
- **解法**：在掃描器 STOP 白名單加入自己人/公開場地/公開平台名（鐘基啟/jacky/彰師大/google/meet…）。
- **預防**：加防呆前先問「這真的是別家客戶的機密，還是公開/自家的字？」白名單要含「自己」。
- **發生於**：2026-06-15 全批寄前掃描。

## 採坑 #5：同公司不同寫法被當成跨客戶
- **症狀**：寄給「陳景崧／宗珀股份有限公司」被擋，理由「含其他客戶『宗珀』」（另一位同事填的是「梁先生/宗珀」）。
- **誤區假設**：名字 token 完全相等才算同一人/同公司。
- **真實原因**：自己人名單存的是「宗珀股份有限公司」，別人填「宗珀」，字串不相等 → 誤判跨客戶。
- **解法**：比對 self 時，剝掉公司後綴（股份有限公司/有限公司/企業/實業/工業/科技…）取「公司核心」再比，同核心＝同公司。
- **預防**：中文公司名比對要正規化後綴；分詞正則也要含**全形**分隔符（`／｜`），否則「人名／公司」切不開。
- **發生於**：2026-06-15 全批寄前掃描。

## 採坑 #6：免費 Gmail 量寄會進垃圾黑名單
- **症狀**：打算一個早上連寄 66 封內容相近、都帶同網域連結的信。
- **誤區假設**：沒超過每日上限就沒事。
- **真實原因**：免費 Gmail 上限 500/天（66 遠低於沒問題），但**真正的雷是「突發爆量 + 無大量寄信歷史」**會觸發外寄垃圾啟發式，輕則進對方垃圾匣、重則帳號限流。Gmail 嚴格大量寄件者規範（SPF/DKIM/DMARC、List-Unsubscribe、投訴率<0.3%）門檻是 5,000/天，我們用不到，但 spike 風險仍在。
- **解法**：分批跨天（每天 30–40 封）、間隔寄（每 2 分一封）、內容有客製不全同、連結分散；對象是填過問卷的溫名單、投訴率低。
- **預防**：新帳號/低寄信歷史帳號做 outreach，先小量暖機、分天、間隔，別一次爆量。
- **發生於**：2026-06-15 排程寄送前（J董喊停要求先查規範）。
