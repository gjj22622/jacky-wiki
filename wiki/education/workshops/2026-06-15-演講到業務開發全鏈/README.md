# 從一場演講到業務開發——AIRUN 端到端獲客全鏈

> 一句話：**先把「成交基礎建設」蓋好，再上台；台上每一頁都對著一個「下一步動作」設計；台下用程式把名單接成可經營的客戶。**
>
> 這份文件把 2026-06-13 彰師大演講這一仗，從演講前建站到演講後業務開發的完整邏輯、流程、自創方法與踩坑，整理成可複製、可傳承的打法。

## 學習目標

1. 看懂「演講＝獲客漏斗入口」這件事該怎麼用基礎建設去接。
2. 能照著四階段（建站前置 → 內容對齊 → 名單接收 → 業務開發）複製一遍。

## 階段 1：演講前——把「成交基礎建設」先蓋好

**核心觀念：演講當天才想怎麼收名單就太遲了。把整條成交動線在上台前架好、壓測過。**

1. **多品牌母站架構**：`airun.tw` 為母站，旗下 明察（政府情報）/ 明腦（企業 AI 導入）/ AI 管理學院（課程）三條產品線；`mingcha.airun.tw`、`mingnow.airun.tw` 子網域用反向代理收斂到母站路徑（`/mingcha/*`、`/mingnow/*`）。
2. **現場成交三路徑**：QR →`go/now`短網址 →`/0613/offer`成交頁；另一條 `/0613-survey` 課後問卷。三路徑壓測（20 連發）通過才上台。
3. **名單收口確定性管線**：前台 `/api/lead`（含限流，演講現場大家共用出口 IP，用 IP+UA hash 細粒度）→ 寫 `leads.json` 持久卷 → 本機 `intake-pull` 每 5 分鐘拉走、建 CRM 案。**站台不跑需要金鑰的東西（gog），重活拉回本機做**。
4. **配套**：藍新金流（1 元測試單真錢驗收）、一對一預約系統（`/book`）、AEO 官網健檢 lead magnet。

> 延伸閱讀：[airun GA4 / 短網址 / UTM 建置](../../../ailab/inbox/2026-05-22-session-airun-ga4-shorturl-utm.md)

## 階段 2：演講中——每一頁都對著一個「下一步」

**核心觀念：簡報不是講完就算，是把聽眾「分類」並「導向動作」的工具。**

1. **內容對齊在地痛點**：簡報「六大場景」（找標案／報價毛利／缺工／老師傅傳承／品質客訴／接班決策）直接對著彰化製造業老闆每天在煩的事，不講抽象 AI。
2. **課後問卷＝需求分類器**：問卷勾選項（公司內訓／免費診斷／一對一／學院通知／只要簡報）本身就是 lead 的**意向分級訊號**，決定後續怎麼接。
3. **現場即時收**：QR 當場掃、當場填，趁熱度最高時把資料留下。

## 階段 3：演講後——把名單接成可經營的客戶

1. **先確認名單真的進系統**（血淚見階段4踩坑）：73 個現場 lead 一度看似「零轉換」，真相是 git bug 沒投影，磁碟其實都在。
2. **確定性貼標分類**：`survey-router` 零 LLM、依勾選把每筆 lead 路由到對應產線（診斷→明察線、內訓→明腦線、學院→waitlist…）。
3. **跟進信產線**（詳見 playbook：[演講後跟進信-人工審核分批寄送SOP](../../playbooks/演講後跟進信-人工審核分批寄送SOP.md)）：
   - 確定性模板（零 LLM、零編造）一人一封綜合信，按勾選組 CTA；
   - **人工逐封審核四軸**：① 原始勾選對不對 ② 稱呼 ③ 內容 ④ 同意寄出；
   - **客製鐵則**：有具體痛點 / 有人脈關係 / 高價值產業 → 跳出模板客製（引用簡報對應頁、給具體建議、誠實框定能力）；
   - **分批寄送**：依風險由低到高（僅簡報→學院→一對一→內訓→診斷）。
4. **缺什麼補什麼**：發現診斷信是空頭支票（不知產業撈不到標案）→ 當場補建 `/diagnosis` 結構化問卷頁接明察產線；明腦無說明頁 → 定位到既有 `mingnow.airun.tw`。
5. **帳戶級彙整**：同公司多人報名（如聚隆纖維 3 位）→ 彙整成「公司級導入」備忘，當談判地圖。

## 階段 4：沉澱出來的自創方法（最值錢的部分）

| 方法 | 一句話 | 為什麼 |
|------|--------|--------|
| **走程式不手刻** | 體檢/回信/診斷一律走有機密邊界護欄的產線，不手做 | 防跨客戶外洩、可重複、可規模化 |
| **反幻覺鐵律** | 報告只放查得到的真實案號/金額，查不到不寫 | 政府情報一旦編造，信任歸零 |
| **自助優先（省 CEO 時間）** | 用說明頁＋預約取代「逐人 20 分鐘解說」 | 老闆不該當客服；頁面替你講 |
| **缺資料去上游撈＋二次快取** | 本機沒有的資料集去 twinkle-hub 撈，用第二次拉回本機 | 不對客戶說「做不到」，也不每次打上游 |
| **觀測管道優先** | 下「沒成果」結論前先驗證「我查的那層是不是壞了」 | 差點誤報零轉換 |
| **寄信合規分批** | 免費 Gmail 500/天、避免突發爆量，分天寄、間隔寄 | 不讓寄件帳號進垃圾黑名單 |

## 採坑點（血淚，獨立詳列）

完整見 [pitfalls：Gmail 量寄與跨客戶洩漏掃描採坑點](../../pitfalls/gmail量寄與跨客戶洩漏掃描-採坑點.md)。這裡先給最致命三條：

1. **「看板沒資料」≠「沒發生」**：73 lead 全在磁碟，卻因 git commit 對 gitignore 路徑失敗沒投影進 read model，差點對外誤報「演講零轉換」。先驗觀測管道，磁碟才是真相。
2. **空頭承諾**：診斷信寫「我幫你撈產業情報」，但根本沒問對方產業／關鍵字——做不出來。改成先問（問卷頁）再做。
3. **別用 import 測有副作用的腳本**：`node -e 'import("寄信腳本")'` 直接觸發真寄迴圈，幸好被洩漏掃描全擋下、0 外洩。要驗證用 `--dry-run`。

## 快速回顧

把演講當獲客漏斗：**上台前蓋好成交動線（建站+名單管線+壓測）→ 台上用內容分類並導向下一步 → 台下用程式把名單確定性接成客戶、人工審核把關品質、分批合規寄出**。所有重活走有護欄的產線、不手刻；缺的補建、查不到不寫、能自助就別佔老闆時間。
