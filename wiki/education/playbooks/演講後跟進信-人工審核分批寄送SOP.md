# 演講後跟進信——人工審核到分批寄送 SOP

## 適用情境

一批活動/演講 lead 已進 CRM，跟進信草稿已用**確定性模板**（零 LLM、零編造、一人一封綜合信、按勾選組 CTA）產好，現在要**人工逐封把關 → 排程合規寄出**。

## 前置條件

- [ ] 草稿在 `outreach/<活動>/*.txt`（每人一封，內含 `To:`/`Subject:` 標頭 + 本文）
- [ ] 有 `index.md` 對照（姓名/email/勾選/檔名）
- [ ] CRM 每筆案可查 `intake.needs`（問卷原始勾選＝審核第一手依據）
- [ ] 寄送走唯一正規通道 `send-deliverable.mjs`（內含跨客戶洩漏掃描 + 寄前驗證）
- [ ] 寄件帳號的每日上限（免費 Gmail＝500/天）

## 步驟

### 1. 建對照表 + 記錄檔 + 佇列
- [ ] 解析 `index.md` × CRM case，建 `.review-map.json`（含每人原始 `intake.needs`）
- [ ] 建 `review-log.md`（記審核標準與逐封決策）、空的 `approved-queue.jsonl`
- [ ] 排序：**風險由低到高**（僅資料/通知 → 導購 → 高承諾），先過最單純的

### 2. 逐封人工審核（四軸）
每封呈現給拍板者：**對象資訊（姓名/公司/來源/狀態/問卷全文）＋稱呼＋回信全文**，逐封確認：
- [ ] **a. 原始勾選對不對**：草稿分類 vs 問卷原始 `intake.needs` 是否相符
- [ ] **b. 稱呼**：無名者→「您好，」；公司名當稱呼可維持；熟人改親近稱呼
- [ ] **c. 內容**：連結正確、無重複段、誠實不誇大
- [ ] **d. 同意寄出**：拍板者說 OK 才進佇列

### 3. 套用客製鐵則（跳出模板的時機）
- [ ] 問卷有**具體痛點/補充** → 引用對應簡報頁 + 給具體可行建議 + 誠實框定能力
- [ ] **人脈關係**（同學/老客戶/前輩/朋友）→ 換熟人口吻、不硬導購；該致謝就致謝
- [ ] **高價值產業**（如營造）→ 多寫兩句、透露 AI Agent 實際動作建立信任
- [ ] **既有/活躍客戶**（已在收報告/談案中）→ **剔除**，走原本的線，別用新手公版

### 4. 排程寄送（合規）
- [ ] 凍結已核可清單為快照（隔離後續新增）
- [ ] 寫定時寄送器：讀佇列、剝 `To:`/`Subject:` 標頭只送本文、走 `send-deliverable`、寄成記 `.sent.json`（斷點不重寄）
- [ ] **合規分批**：每天 ≤ 上限的一小部分（這次 30–36/天）、間隔寄（每 2 分一封）、**分多天避免突發爆量**
- [ ] systemd timer 觸發（注意 `OnCalendar` 要加 `Asia/Taipei`，預設是 UTC）

### 5. 寄前最終驗證
- [ ] 全批跑跨客戶洩漏掃描（leak scan）＝乾淨
- [ ] **查重**：email 去重（含 gmail 點/大小寫正規化、跨網域同 local-part）
- [ ] dry-run 確認佇列筆數正確、標頭剝除正常

## 異常情況分支

| 情況 | 處理 |
|------|------|
| leak scan 擋下自己人/場地/公開產品名（鐘基啟/彰師大/Google Meet） | 加 STOP 白名單——是公開/自家字，非他人機密 |
| leak scan 擋同公司不同寫法（宗珀 vs 宗珀股份有限公司） | 剝公司後綴取核心比對 self |
| 收件 email 疑似 typo（gamil.com） | 停下確認，修正成 gmail.com 或跳過（gamil 是釣魚網域，會外洩） |
| 既有活躍客戶混進名單 | 剔除、CRM 標 `excluded-from-<活動>`、decisions_log 留痕、reproject |
| 測試資料（example.com / 「分流測試」） | 剔除、標 `test-record` |
| 草稿承諾做不到的事（不知產業就說幫撈情報） | 改成「先問再做」——補建結構化問卷頁收必要輸入 |

## 驗收標準

- [ ] 每封都有拍板者明確「OK」並記入 `review-log.md`
- [ ] 佇列無重複 email、全過 leak scan
- [ ] 寄送器 dry-run 筆數正確、可斷點續傳
- [ ] 分批排程每天量 ≪ 帳號上限、間隔寄、跨多天
- [ ] 剔除案都已在 CRM 留痕
