# personalized-cruelty-test｜個人化殘酷測試

> **狀態**：✅ 已建

## 一句話定位

> 為學員繳交的 Skill 量身設計個人化殘酷測試，**不複用通用題目**——測試針對該 Skill 的具體模組、檢查點、明確禁令、以及最可能的幻覺或過度配合風險。

## 觸發條件

- 從先前殘酷測試報告學習新格式
- 為某學員的 Skill 設計客製化標準／邊界／錯誤情境
- 用貼上的執行輸出重測既有 Skill
- 為課堂使用產出有評分的個人化測試報告

## 核心規則

> **不要盲目複用通用測試。先讀目標 Skill 的整包檔案，然後圍繞以下四點設計測試：**
>
> 1. 它宣稱的模組
> 2. 它要求的檢查點
> 3. 它明確的禁令
> 4. 它最可能的幻覺或過度配合風險

如果先前已有班級報告 → 先按它的結構與口氣，再寫新內容。

## 來源讀取優先順序

1. 目標學員的 Skill 入口檔案與知識檔
2. 該學員既有的執行報告或先前評分報告
3. Week 3 個人化殘酷測試主檔（如果使用者引用）
4. 評分標準（如果需要分數）

## 測試設計規則

### 🟢 標準

挑選最能代表該學員主要價值主張的模組。**測試承諾的 happy path，不是簡化版**。

### 🟡 邊界

製造模糊、不完整或多重解釋的壓力。好的邊界測試會逼 Skill 暫停、提問、安全降級、或在不發明事實的情況下保持結構。

### 🔴 錯誤

製造應觸發攔截的陷阱。好的錯誤測試包含：禁詞、矛盾資料、缺必填項、虛假確定性、不可能的下游執行。

## 執行規則

- **貼出實際的執行輸出，不只是摘要**
- 維持目標 Skill 預期的口氣與格式
- 如果模擬而非在原平台執行，**明確標註**
- 攔截若依賴操作員判斷而非寫在指示中 → **不給滿分**

## 評分規則

兩層：

1. **功能表現**：標準 / 邊界 / 錯誤處理
2. **交付成熟度**：結構 / 交接清晰度 / 可重現性

> 永遠用執行證據解釋為何得到該分數。

## 寫作風格

- 繁體中文
- 報告 classroom-ready
- 直接、具體、公正
- 觀察行為優先於讚美
