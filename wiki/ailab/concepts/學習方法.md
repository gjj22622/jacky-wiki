# 學習方法

> **核心信念**：學新工具不是讀文檔，是用真實任務試錯。
> **節奏**：48 小時上手 → 2 週分流 → 1 個月決策（去留／納入工具棧）。

---

## 三階段學習 SOP

### Phase 1：48 小時上手（broad scan）

**目標**：知道這工具能做什麼／不能做什麼，不求精通。

| Day | 動作 |
|---|---|
| Day 0 | 找 1-2 篇官方文檔關鍵章節（不全讀），看「能做什麼」 |
| Day 1 | 跑 3 個 demo（官方範例不算）：用真實素材，跑 3 個迥異任務 |
| Day 2 | 寫一份 [實踐捕手協定](實踐捕手協定.md) 事件，記錄初步印象 |

**成功判準**：能用一句話講清楚這工具的「核心定位」與「明顯短板」。

### Phase 2：2 週分流（comparative use）

**目標**：搞清楚什麼任務該用它、什麼任務不該用它。

操作：把日常任務刻意分流給多個工具做，並記錄結果：

| 任務類型 | 用 A 工具 | 用 B 工具 | 比較 |
|---|---|---|---|
| 跨檔重構 | Claude Code Sonnet | Codex CLI | 哪個漏改少？ |
| 文案產出 | Claude Opus | ChatGPT GPT-5.4 | 哪個風格更穩？ |
| 圖像生成 | gpt-image-2 | Canva MCP | 哪個成本/品質好？ |

**成功判準**：能寫出「**任務 → 工具**」的選擇規則（落地到 [模型選擇心法](../tools/模型選擇心法.md)）。

### Phase 3：1 個月決策（keep / drop / niche）

**目標**：決定這工具在工具棧的位置。

三種落點：

| 決策 | 條件 | 動作 |
|---|---|---|
| **Keep（納入主工具棧）** | ≥ 3 個任務上明顯優於替代 | 寫進 [工具棧索引](../tools/工具棧索引.md)、建獨立 tool 頁 |
| **Niche（保留特定場景）** | 只在 1-2 個場景明顯優 | 在 [模型選擇心法](../tools/模型選擇心法.md) 註記「**只在 X 場景用**」 |
| **Drop（不納入）** | 沒明顯優於既有、學習成本高 | 寫一份 [實踐捕手協定 type=失敗] 事件，標 `maturity: 已穩定`（已決定不用），日後不重複試 |

## 學新模型 vs 學新工具的差別

### 學新模型（GPT-6、Claude 5、Gemini 3）
- 重點：**能力差異**，不是介面
- 方法：在**既有 IDE / CLI** 切換 model 跑同樣任務
- Phase 1 縮短到 24 小時（介面沒變）

### 學新工具（新 IDE、新 CLI、新 MCP）
- 重點：**整合方式**，不只是能力
- 方法：建獨立 sandbox，從 0 跑到底
- Phase 1 完整 48 小時（介面要熟）

## 三種「學習觸發點」

什麼時候該學新工具？以下信號出現 ≥ 2 個：

| 信號 | 例子 |
|---|---|
| 既有工具明確不夠用 | Claude 跨檔重構漏改 → 學 Codex |
| 新工具解決明確新需求 | Jwood 商品圖大批量 → 學 gpt-image-2 |
| 工具圈出現結構性變化 | MCP 標準成熟 → 全面學 MCP 生態 |

### 不該學的時機
- ❌ 「聽說新功能很潮」（純追新）
- ❌ 「benchmark 第一」（benchmark ≠ 實用）
- ❌ 「同行都在用」（同行的場景不一定跟你一樣）
- ❌ 「老板／客戶要求」（先談清楚要解決什麼問題，再決定要不要新工具）

## 學習過程的記錄義務

每學一個新工具／新模型，**必須產出**：

| 產物 | 落點 |
|---|---|
| Phase 1 的初印象事件 | `ailab/inbox/` |
| Phase 2 的分流規則 | 直接寫進 `ailab/tools/模型選擇心法.md`（更新而非新增）|
| Phase 3 的去留決策 | `ailab/log/<日期>-<工具名>決策.md`（重大）或 `ailab/inbox/` 結案註記（一般）|

> **不寫 = 沒學過**。否則 6 個月後忘了當時為什麼選/不選，又從頭來一遍。

## 實際案例：學 Codex CLI（2026-04）

| Phase | 動作 | 結論 |
|---|---|---|
| Phase 1 | 跑 3 個跨檔重構 demo | 結構性重構比 Claude 穩 |
| Phase 2 | 雙跑 Claude Code + Codex CLI 兩週 | 「跨檔結構重構 → Codex；單檔 + 對話 → Claude」 |
| Phase 3 | 決策：Niche（保留特定場景）| 寫進 [模型選擇心法](../tools/模型選擇心法.md)、加 codex plugin |

## 「找文檔」的反原則

### 不要這樣
- ❌ 從頭讀官方完整文檔（3 小時讀完，沒實作沒記住）
- ❌ 看 YouTube 教學影片（速度慢、過時快）
- ❌ 找「最佳實踐」清單（跟你的場景不一定符合）

### 該這樣
- ✅ **找 1-2 篇關鍵章節**：通常是「Quickstart」+「Concepts」兩節，看完就跑
- ✅ **官方 SDK 的範例 repo**：直接 clone 然後改成你的場景
- ✅ **changelog**：看「最近 3 個版本改了什麼」，比讀全文檔快很多
- ✅ **真實任務驅動**：拿手邊一個真實任務當試金石，邊做邊查

## 多場景對照律（呼應演化元方法）

> 學單一工具最大的盲點是「樣本不夠」。

**規則**：學新工具時，**至少在 2 個性質迥異的場景跑**：

- 双云客戶任務（高商業壓力、需穩）
- Jwood 個人專案（多模態、長期）
- NCHU 學術（嚴謹度高、可追溯）

不同場景測完才看得到工具的真實能耐。

## 學習成本的估計

| 工具類型 | Phase 1 | Phase 2 | Phase 3 | 總成本 |
|---|---|---|---|---|
| 新模型（介面同）| 4-6h | 並行使用無額外成本 | 1h 寫決策 | ~10h |
| 新 IDE 工具 | 8-12h | 並行需額外切換成本 ~5h/週 | 2h 寫決策 | ~30h |
| 新 MCP / 新 SDK | 4-8h | 自然累積 | 1h 寫決策 | ~15h |
| 新 CLI 框架 | 12-20h | 5h/週 切換成本 | 3h 寫決策 | ~50h |

> Jacky 經驗：每個月留 ~30h 給「學新工具」的預算是健康的。超過就是被工具帶著跑，不到就會 fall behind。

## 與 AGENTS A 階段的對照

> AGENTS A（Analyze）= 拆人的工作給 AI；本頁 = 拆 AI 工具的能耐給人。
> 兩者都是「**把模糊的能力／工作顯性化**」——同樣的元動作、不同的對象。
