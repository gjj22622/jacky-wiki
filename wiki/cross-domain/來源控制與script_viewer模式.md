# 來源控制與 script_viewer 模式

> AGENTS 流程裡，**最重要的品質控制與來源控制工具**。
> 從 NCHU 第三次專討（2026-05-13）的 `script_viewer.html` 萃取的範式。

---

## 一句話定位

**script_viewer 模式 = 事前可追溯性的視覺化工具**——把產出（講稿／文案／報告）與來源（文獻／品牌資料庫／知識庫）並列，讓每一個引用都可以**點擊跳到原文**，從根源杜絕幻覺。

> 殘酷測試是**事後**找錯；script_viewer 是**事前**讓引用無法造假。
>
> 兩者一起構成 AGENTS E 階段的雙軌品管。

---

## 為什麼需要這層工具

### 真實事故：73% 作者名幻覺率（2026-03-12）

第三次專討準備期間，Jacky 檢查 26 個帶作者名的檔案，發現 **19 個錯誤（73% 錯誤率）**，2 個 PDF 的內容與檔名不一致。

事故根因：
- AI 在文獻檢索時會編造第一作者姓名
- 人工核對成本高（要每篇翻 PDF 第一頁）
- 沒有工具讓「引用 → 原文」一鍵驗證

### v1.0 的 E 階段缺口

AGENTS v1.0 的 E 階段定義成「殘酷測試」單一手段（🟢🟡🔴 三色情境）——**只能事後檢測**。但學術／高合規場景需要的是 **事前杜絕**：在產出階段就讓引用必須對應到白名單。

**雙軌品管模型**

| 軌 | 名稱 | 時機 | 工具 |
|---|---|:---:|---|
| 軌 1（事前） | **來源可追溯性** | 產出當下 | script_viewer 模式 + REFS 白名單 |
| 軌 2（事後） | **殘酷測試** | 產出之後 | 三色情境 + 通過率指標 |

## script_viewer 三大關鍵設計

### 1. 兩欄布局（左產出 70% + 右來源 30%）

```
┌──────────────────────────┬─────────────┐
│                          │  REFS PANEL │
│  講稿 / 文案 / 報告        │  (sticky)   │
│                          │             │
│  ...Lin et al. <cite>05</cite> 提出      │  [01] Xu      │
│  Sentinel-2 + XGBoost...  │  [02] Shang  │
│  R²=0.82 <cite>03</cite>  │  [03] Ali    │
│                          │  [04] Pell.  │
│                          │  [05] Lin    │
│                          │  ...         │
└──────────────────────────┴─────────────┘
                              ↑
                    sticky，永遠在右側可見
```

左 70% 主內容固定，右 30% 用 `position: sticky` 跟著捲動——讀到哪一段，右邊隨時可查 14 篇白名單。

### 2. REFS 白名單對照表（hard-coded）

引用編號被**鎖定**在 JS 內的 REFS 物件，每個編號對應：

| 欄位 | 內容 |
|---|---|
| `author` | 「Lin et al. 2025」（必對應 PDF 第一頁） |
| `ch` | 「Ch4」（章節定位） |
| `file` | 「05_Ch4_Lin_2025_中文翻譯.md」（實際檔名） |
| `desc` | 「GEDI + MGWR，RMSE 40.756→13.934」（一句話描述） |

> **白名單是硬邊界**：講稿任何引用都必須對應到這個表，否則無法 hover、無法跳轉，立刻暴露為「未受控引用」。

### 3. cite 標籤 → tooltip → 跳原文（三段式互動）

```html
<cite data-num="05">Lin et al.</cite>
```

- **hover**：顯示 280px tooltip（作者 + 一句話描述）
- **click**：用 `encodeURIComponent` 處理中文路徑後 `window.open` 對應的中文翻譯 .md
- **離線可用**：純 file:// 協定，不需 server，可寄給指導老師直接點開

## 跨場景應用

script_viewer 模式不只學術——任何「產出引用 → 來源資料庫」結構都適用。

### 學術場景（NCHU 已實作）
- **左**：第三次專討講稿 V3
- **右**：14 篇核心文獻白名單
- 用途：演講前每個引用都能跳 PDF 原文驗證

### 行銷場景（双云可建）
- **左**：客戶貼文文案
- **右**：客戶品牌資料庫白名單（產品定位、調性、禁詞、上線資訊）
- 用途：每個事實宣稱都能跳到品牌資料的具體段落

### 木藝場景（Jwood 可建）
- **左**：YouTube 影片講稿（穴道按摩教學）
- **右**：穴道知識來源白名單（解剖位置、按摩手法依據、安全警告）
- 用途：避免影片中提到的醫療資訊憑空編造

### 教學場景（TBSA 可建）
- **左**：5 大表單填寫範例
- **右**：洪敏莉初階教材 + 鄭沂珊進階教材白名單
- 用途：示範案例的每個論點都能跳到教材原句

## 與 AGENTS 各階段的整合

| AGENTS 階段 | script_viewer 模式的角色 |
|---|---|
| A 拆解 | 拆解時就決定「白名單範圍」（這個產出能引用的來源邊界） |
| G 生成 | 寫 cite 標籤時必須對應白名單，否則 viewer 顯示空白 |
| **E 評測** | **事前品管**：上場／發布前用 viewer 巡視所有引用一次 |
| N 串聯 | viewer 是入口導覽頁的補強層（[入口導覽頁 SOP](入口導覽頁SOP.md)） |
| T 追蹤 | 修稿時新版 viewer 與舊版 diff，可看出引用變動 |

---

## 製作流程（建議 SOP）

1. **先建白名單**——產出開始前先決定「合法來源清單」（學術 14 篇、行銷品牌資料 N 段、教學教材 N 章）
2. **產出時用 cite 標籤**——每個引用都標 `<cite data-num="XX">`
3. **REFS 物件 hard-code 在 viewer JS**——拒絕動態載入（避免來源被竄改）
4. **viewer 兩欄 + sticky right panel**——複製博二實例的 CSS 即可
5. **encodeURIComponent 處理中文路徑**——避免 file:// 協定打不開中文檔名
6. **離線測試**：直接雙擊 HTML 確認所有引用可跳

---

## 反模式（不要這樣做）

| 反模式 | 問題 | 正確做法 |
|---|---|---|
| 引用直接寫 `(Lin, 2025)` 不標 cite | viewer 抓不到，無法驗證 | 用 `<cite data-num="05">Lin et al.</cite>` |
| REFS 從外部 JSON 動態載入 | 來源可被竄改、離線不可用 | hard-code 在 viewer 同檔案內 |
| 把全部文獻塞進 REFS（>30 篇） | tooltip 與右欄變嘈雜，失去「核心白名單」意義 | 嚴格篩選 10-15 篇核心 |
| 引用沒對應到白名單仍可發布 | viewer 沒攔截，幻覺仍可能漏出 | 上場前用 grep 比對 cite num vs REFS keys |
