# Week 1｜拆 — 用 Agent 思維重新設計你的工作

> **時長**：2 小時
> **對象**：双云行銷團隊成員（已有 AI 應用基礎）
> **課後產出**：每人 1-5 份 Agent 說明書 + Dataroom 結構 + 調教日誌第一版
> 來源：`02_課程架構與教案/00_Week1_課程架構_拆.md`（2026-04-29 補建）

---

## 本週核心命題

> **「問題不是要不要用 AI，而是你的工作方式還能撐多久？」**

Week 1 的任務只有一件事：讓每個人完成思維翻轉——**從「找個工具用」變成「把每個流程都拆解給 Agent 做」**。說明書是這週唯一的產出，工具只是驗證說明書的載體。

---

## 課程架構（8 模組 / 120 min + 20 min 練習）

### 模組 1｜觀念翻轉：為什麼是 Agent，不只是 AI？（20 min）

**1-1 打開真實處境（10 min）**

不急著教工具，先讓大家感受到「現在的工作方式確實有問題」。三個切入點：

- **客戶期待在變** — 客戶要的不再是一份企劃，而是持續優化的系統
- **產能瓶頸** — 重複性工作佔 60%+ 時間，案量在增加但人力沒變
- **競爭者在加速** — 別的公司已經在用 Agent 搶案

**1-2 AI 工具 vs AI Agent（10 min）**

| | AI 工具 | AI Agent |
|---|---|---|
| 互動方式 | 你問它才答 | 主動參與流程 |
| 記憶 | 每次從零開始 | 帶記憶、帶判斷 |
| 角色 | 被動的問答機器 | 你的數位同事 |
| 特性 | 沒有角色與職責 | 有角色、有職責 |

**AI First 核心精神**

> 「每一個流程，先假設 AI Agent 來做，再決定哪些需要人介入。」
> 這不是「找到一些可以用 AI 的場景」，而是**全面重新設計工作方式**。

---

### 模組 2｜Agent 說明書：你的搬家保險（15 min）

> **核心觀念：工具會變，說明書不變。**

今天在 Claude 建好的 Agent，明天換到 Gemini，**不用重建，只需要「搬家」**。投資在設計上，不要投資在工具上。

#### 說明書五大要素

```
┌─────────────────────────────────────────────────────┐
│  1. 角色定位   — 你是誰？擅長什麼？                  │
│  2. 觸發條件   — 什麼時候該啟動？                    │
│  3. 輸入／輸出 — 吃什麼資料、吐什麼結果？格式要求？  │
│  4. 判斷邏輯   — 遇到選擇時怎麼決定？                │
│  5. 邊界定義   — 什麼不做？什麼交給人？              │
└─────────────────────────────────────────────────────┘
```

> Week 2 會把這五大要素升級為 [9 區塊 Agent 設定檔](../concepts/AGENTS知識體系.md#g--generate生成建構)。

---

### 模組 3｜三大主流工具（15 min）

> **核心觀念：平台只是載體，同一份說明書在三個平台都能用。**

| 平台 | 定位 | 双云用途 |
|---|---|---|
| **Gemini Gem** | Google 生態整合（Gmail / Docs / Meet） | Google 生態串接 |
| **Claude Skill** | 結構化推理專家（長文 + 多步驟推理） | **核心思考與推理（双云 Agent 體系核心）** |
| **OpenClaw** | 開源 × 本地 × 自由（50+ 整合 + 瀏覽器自動化） | 跨平台自動化、WhatsApp / Telegram 操控 |

**同一份說明書 → 三個平台**

```
Agent 說明書（設計圖）
     ↓
Gemini Gem   → 貼入 Gem 設定頁面的指令區 + 知識庫
Claude Skill → 存為 SKILL.md，定義角色 / 觸發 / 輸出
OpenClaw     → Custom Skill 部署，本地 + 綁 WhatsApp
```

> **教學意圖**：不要陷入「哪個平台比較好」的選擇困境。重點是「說明書先行」——設計好了，去哪都一樣。

### 模組 4｜Agent 協作架構（15 min）

#### 双云的三層 Agent 架構

```
大腦 / 顧問                執行層                  審查 / 質疑
────────────           ────────────           ────────────
  Jacky Agent    →→→  各功能 Agent    →→→  Jacky 問問題
  策略判斷            企劃 / 提案 /           蘇格拉底式提問
  方向建議            客戶溝通 Agent…         盲點檢查
  決策者角色                                  品質把關
```

#### Agent 協作工作流（以 AI 備課流程為例）

1. **執行前** → 問 Jacky Agent 要方向（分析學員問卷、給 Persona 與備課方向）
2. **執行中** → 備課 Agent 產出提案初稿
3. **執行後** → Jacky 問問題 Agent 蘇格拉底式審查，找盲點
4. **修正** → 執行 Agent 根據審查意見修正
5. **再審查** → 確認品質達標後交付

> **教學意圖**：讓學員看到「一個完整的工作流程，不是一個 Agent 做完，而是多個 Agent 接力」。為 [Week 4（串）](Week4串.md) 打下觀念基礎。

### 模組 5｜人也要建 Agent：把關鍵人物數位化（10 min）

#### 双云的「人物 Agent」地圖

| 人物 | Agent 記憶的內容 |
|---|---|
| 客戶窗口 | 溝通風格、回覆偏好、在意的 KPI |
| 客戶老闆 | 決策邏輯、在意的事、過往踩過的雷 |
| 合作夥伴 | 合作模式、利益考量、技術能力邊界 |
| 競爭者 | 策略模式、優劣勢、近期動態 |

**原料從哪來**：會議紀錄 / Email 往來 / 提案修改歷史 / LINE 對話 / 過往合作踩坑紀錄

**怎麼用**

- 準備提案前 → 讓「客戶老闆 Agent」模擬對方會問什麼
- 寫信前 → 讓「客戶窗口 Agent」預判對方反應

> **教學意圖**：Agent 應用邊界比想像更廣——**不只是「做事」，還可以「模擬人」**。同時開始讓大家思考「我的 Dataroom 原料在哪」。

---

### 模組 6｜Agent 間的溝通：多窗口混用（10 min）

> **標準化輸出格式 = Agent 之間的接力棒**——A 的輸出，直接變成 B 的輸入。沒有標準格式，Agent 就無法串接。

**多窗口實作（提案流程）**

```
提案初稿（執行 Agent 產出）
       ↓
  ┌──────┬──────┬──────┐
  行銷  双云  老闆
  經理  Agent Agent
  Agent（品管）（策略）
  （創意）
  └──────┴──────┴──────┘
       ↓
  大腦 Agent 彙整三方意見
       ↓
  問問題 Agent 最終審查
```

**OpenClaw 跨平台整合**：可透過 WhatsApp / Slack 多管道同時操控不同 Agent，最貼近真實工作場景。

---

### 模組 7｜怎麼建立 Agent：三種餵養方式（10 min）

| 方式 | 適合場景 | 操作重點 |
|---|---|---|
| **手動對話輸入** | 你自己的隱性知識 | 直接跟 AI 對話，把腦中的判斷邏輯、潛規則灌進去 |
| **會議紀錄輸入** | 師父知識萃取 | 把逐字稿、錄音摘要、Email 餵 AI，從真實互動萃取知識 |
| **反覆調教** | 使用後優化（**最關鍵**） | 實際使用後針對不對的地方修正，建「調教日誌」追蹤改善 |

> **銜接關鍵**：三種方式不是選一個，而是循環用。**先手動輸入建初版，再用調教日誌持續優化。**

### 模組 8｜Dataroom：讓 Agent 變聰明的關鍵（15 min）

> **Dataroom 的本質：Agent 的記憶體。記憶體越大、越乾淨，Agent 越聰明。**

#### 實際 Dataroom 結構（AI 教師研習營範例）

```
AI教師研習營 Dataroom/
├── 01-課程背景/    ← 教育部政策文件、學校需求調查
├── 02-受眾資料/    ← 教師背景、AI 熟悉度、痛點訪談
├── 03-知識庫/      ← 教學案例、AI 工具教學、FAQ
├── 04-對話範例/    ← 過往 Q&A、常見誤解、應對話術
└── 05-決策邏輯/    ← 課程推薦規則、程度判斷標準
```

#### Dataroom 設計四原則

1. **結構化分類** — 不是丟一堆檔案，是按 Agent 需要的「思考維度」分類
2. **持續更新** — 每次互動、每次修正，都要回寫到 Dataroom
3. **品質 > 數量** — 10 份精準對話範例 勝過 100 份雜亂文件
4. **可追溯** — 每份資料標注來源和日期，方便回查和清理

#### 客戶 Dataroom 四步建立流程

```
Step 1 盤點原料來源
       → 會議紀錄 / Email / LINE 對話 / 提案歷史
       → 客戶官網 / 產業報告 / 社群動態

Step 2 建立資料夾結構
       → 客戶背景 / 窗口人物誌 / 專案歷史
       → 決策邏輯 / 溝通偏好 / 踩坑紀錄

Step 3 餵養 Agent
       → 將結構化資料匯入 Skill / Gem / OpenClaw
       → 設定觸發條件與輸出格式

Step 4 持續迭代
       → 每次互動後回寫 Dataroom
       → 定期清理過時資料、補充新資訊
```

### 現場練習｜建你的第一個 Agent + Dataroom（20 min）

> **Jacky 的提醒**：先求有，再求好。第一版不完美是正常的，重點是「**開始**」。

| Step | 時間 | 動作 |
|---|---|---|
| 1 | 2 min | 選一個工作場景（最常重複做的事，例如：回覆客戶詢問、整理會議紀錄、製作報價單） |
| 2 | 5 min | 建立 Dataroom 結構（列出資料夾、原料、來源） |
| 3 | 8 min | 寫出 Agent 說明書（套用五大要素） |
| 4 | 5 min | 在平台上建立並測試（Claude / Gem / OpenClaw 任選；測 3 種情境：標準／邊界／例外） |
