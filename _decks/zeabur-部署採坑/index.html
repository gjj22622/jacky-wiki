Zeabur CLI 部署 · 採坑點

AI 大腦團隊上線那晚
踩的 5 個雷

內部團隊分享 約 10 分鐘 2026-06-22
成員、key、環境變數全都備好了——卻卡在「最後一步部署」整整一小時。
這 5 個雷全是那晚真實踩出來的。先看雷，再記三條鐵則。
01
#1

從錯的目錄 deploy，
服務直接變 404

▼ 現場 · 錯誤做法
症狀
deploy 回報「成功」，但帶正確憑證打 API 卻 404，連原本好的 endpoint 都不見了。
# 在 tools/members/ 跑（錯！） npx zeabur deploy --service-id … Service deployed successfully # 但 POST /order → 404 not_found
誤區假設
以為 deploy 認的是 --service-id，在哪個資料夾跑都一樣。
▲ 真相 · 正確做法
真實原因
zeabur deploy 上傳的是當前工作目錄（CWD）。從 tools/members/（只有一支 CLI）跑，等於把錯內容當服務上傳、把好服務蓋掉。
解法
先 cd 到該 service 的源碼目錄再 deploy。
cd services/fulfillment # 含 Dockerfile 那層 npx zeabur deploy --service-id … --environment-id …
預防
部署前一律先 pwd 確認；把「cd 對目錄 + deploy」封進腳本。
02
#2

改了 env，
restart 不會生效

▼ 現場 · 錯誤做法
症狀
CLI 更新環境變數成功、restart 也成功，但服務行為沒變——新成員 key 還是 401。
誤區假設
以為「restart 一下就會重讀新環境變數」。
variable update … MEMBERS_JSON_B64=… ✓ service restart ✓ # 新 key 仍 → 401 unauthorized
▲ 真相 · 正確做法
真實原因
服務只在啟動時讀一次 env；restart 用的是舊部署的 env 快照。
redeploy 又只給 git 綁定服務（本機上傳型回 CANNOT_REDEPLOY_INPLACE）。
解法 + 預防
改完 env → 從源碼目錄跑一次成功的 deploy（重新上傳＝把新 env 烘進新部署）。
# 心法 改 env ＝ 要 deploy，不是 restart
03
#3

把「上傳逾時」
當成部署失敗

▼ 現場 · 錯誤做法
症狀
deploy 卡在 upload 逾時，以為壞了，反覆改設定、瞎查。
failed to prepare upload … context deadline exceeded
誤區假設
逾時 ＝ 我哪裡設錯了。
▲ 真相 · 正確做法
真實原因
那是 Zeabur 上傳端的網路 transient，跟你的設定無關。env 變數那步通常已成功，只是上傳沒過。
解法
直接重試（多跑幾次）或換較穩網路。
預防
看到 deadline exceeded 先重試 2–3 次，再懷疑設定。
04
#4

名冊/憑證不在 git，
多機「各說各話」

▼ 現場 · 錯誤做法
症狀
想讓另一台機器（Linux）也能部署/改成員，但那台的名冊跟主機不一致。
誤區假設
以為 git pull 會把一切同步過來。
▲ 真相 · 正確做法
真實原因
名冊真相檔（members.json）與金鑰刻意 gitignored（不能進公開 repo）→ 不跟著 git 走，兩台各改就分歧。
解法
指定一台當「名冊真相機」只在那台改成員；其他機要用時，經安全管道（非 git）複製最新檔。
預防
程式碼走 git；機密與狀態走帶外管道。明確誰是真相機。
05
#5

驗證時把 401 跟 404
混為一談

▼ 現場 · 錯誤做法
症狀
部署後打 API 失敗，分不清是「沒生效」還是「部署壞了」，亂改一通。
預防 · 暖機
剛 deploy 完服務重啟數十秒會短暫 4xx，正常。用 curl --retry 等就緒，別用 shell sleep。
▲ 三態判讀 · 一眼定位
回傳碼怎麼讀
401 帶 key 卻 unauthorized → 名冊沒 deploy 生效（雷 #2）
404 帶對 key 卻 not_found → 上傳錯目錄（雷 #1）
200 · fulfilled:false → 認證 OK，只是這題沒命中（不是錯）
200 · fulfilled:true → 完全正常 ✓
06
收束 · 帶走這個就夠

三條黃金鐵則

01

部署前先 pwd

deploy 上傳當前目錄。一定先 cd 到 service 源碼夾（含 Dockerfile）再 deploy，否則服務變 404。

02

改 env ＝ deploy
不是 restart

服務只在啟動讀 env。restart 用舊快照、redeploy 需 git 綁定——改完變數必須重新 deploy 才生效。

03

逾時 ＝ transient
重試就好

prepare upload deadline exceeded 是 Zeabur 網路問題，不是你設錯。重試 2–3 次通常就過。

外加兩條：名冊/憑證走帶外管道不走 git（多機真相機）· 驗證時 401 ≠ 404 別亂改。
