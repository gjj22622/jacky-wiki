# 双云多品牌獨立應用架構

## 核心決策

双云 AI 行銷部不是把所有品牌資料塞進同一個大系統，而是：

```text
双云 AI 行銷部 Command Center
→ 管理多個可獨立運作的 Brand App
→ 每個 Brand App 都有自己的品牌腦、任務、成員環境、品管規則、素材與成效資料
```

這個架構的目的，是讓每個甲方品牌能保有獨立、機密、可控的營運環境，同時又能被双云中控台協作、調度、review 與追蹤。

## 為什麼不能全部塞進同一個品牌工作區

每個品牌都會有自己的：

- 老闆意志與決策邊界
- 商品、服務與營收目標
- 客群、痛點與成交情境
- 視覺系統與語氣規則
- Meta / Google Ads / LINE / 電商資料
- 資深成員操作脈絡
- 新人可執行範圍
- reviewer 品管規則

如果所有品牌共用一個內容池、prompt、任務池或素材規則，一個品牌的錯誤資料可能污染其他品牌，導致機密外洩、策略混淆或錯誤輸出。

## 架構分層

### 1. 双云 Command Center

中控台負責看跨品牌狀態，而不是混放所有品牌細節。

它應該管理：

- 目前有哪些品牌正在營運
- 哪些成員被分配到哪些品牌
- 哪些新人正在進入哪個品牌
- 哪些任務卡住
- 哪些 submission 等 reviewer
- 哪些品牌有營收貢獻訊號
- 哪些品牌需要資深成員介入

### 2. Brand App

每個品牌都是一個邏輯上獨立的應用。

一個 Brand App 至少包含：

- Brand cockpit
- Brand brain
- Product brain
- Customer brain
- Visual brain
- Daily tasks
- Submissions
- Reviews
- Trace logs
- Revenue signals
- Senior member activity
- Channel modules：ecommerce / Meta / Google Ads / LINE

### 3. Member Workspace

每個成員也應有自己的獨立環境。

成員環境包含：

- 被指派的品牌
- 被指派的任務
- 可操作的 channel
- 個人訓練進度
- 個人 trace history
- 可見的 reviewer queue

成員可以協作，但協作必須透過明確的品牌任務、review、comment 或 senior member activity，不應因為能進入平台就看到所有品牌資料。

## 資料隔離原則

所有品牌營運資料都必須帶 `brandId`。

```text
BrandTask.brandId
BrandBrainNode.brandId
Submission.brandId
Review.brandId
RevenueSignal.brandId
SeniorMemberActivity.brandId
ChannelAccount.brandId
VisualAsset.brandId
```

所有成員操作資料都應有明確的成員或權限欄位。

```text
memberId
assigneeId
ownerId
reviewerId
actorId
```

底線：

> 沒有 `brandId` 的品牌資料，不應進入正式品牌營運流程。

## Skill / Prompt 隔離原則

Prompt context 應該按範圍組合：

```text
Global 2clouds Method
+ Member Role / Permission Context
+ Brand-specific Skill
+ Brand-specific Brain Nodes
+ Task Context
+ Reviewer Rules
```

品牌專屬 skill 不得在其他品牌任務中自動載入。木酢寵物達人的品牌定位、視覺規範、客戶資料、廣告學習，不應污染其他甲方品牌。

## 與既有 AI 行銷部模型的關係

本頁延伸自 [双云 AI 行銷部 — 定義與操作指南](AI行銷部定義.md)。

原本的「每個客戶一套 4 Agent 委員會」在平台化後，應升級成：

```text
每個品牌一個 Brand App
每個 Brand App 內含自己的 4 Agent 委員會、品牌腦、任務流、品管流、營收訊號
双云 Command Center 只做跨品牌調度與總覽
```

## 開發方向

2clouds-agents 專案下一階段應優先建立：

```text
Multi-brand Command Center + Isolated Brand App Shell
```

驗收標準：

- 可以看到品牌列表
- 可以進入單一品牌工作區
- 單一品牌工作區有品牌腦摘要、日常任務、資深成員活動、營收訊號
- domain model 中的品牌資料都以 `brandId` 隔離
- 成員環境能以 member / assignee scope 隔離
- 品牌 skill / prompt context 不跨品牌共用

## Jacky 的架構要求

Jacky 明確要求：

> 双云 AI 行銷部是多個獨立應用結合的中控。每個應用都可以獨立運作；每個成員都有各自的獨立環境；品牌間保有機密獨立運作，但又可以在双云中控下協作。遇到不確定的架構問題，要回到 Jacky Wiki 討論出最佳方案。

這是後續平台架構設計的上位原則。
