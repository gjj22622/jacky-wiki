# n8n + LangSmith 落地計畫

> 對「業界主流 Agent 平台清單」的評估結論落地版。
> **背景**：七個外部平台中，只有 **n8n**（串接層）與 **LangSmith**（觀測層）通過 jos 哲學檢驗（廠商中立、可攜、可還原）。其餘五個（Vertex / Bedrock / Cloud Run / CrewAI / Zapier）判定為「請你搬進別人的 OS」，不採用。
> **判準（一條）**：這個平台是**裝進**我的 jos 作業系統，還是要我**搬進**別人的封閉 OS？前者才落地。

---

## 為什麼是這兩個（一句話定位）

| 平台 | 落在 jos 哪層 | 補的缺口 | jos 相容性 |
|---|---|---|---|
| **n8n** | 工具層（自架在環境層）| Jwood / 双云的**跨 SaaS 自動化串接** | 🟢 開源可自架、可寫進 environment/、`/移機` 可還原 |
| **LangSmith** | 觀測層（外掛双云平台）| 双云腦+手跑 20 客戶的 **trace / 評估 / 抓錯** | 🟡 SaaS 觀測層、不碰 OS 核心、低耦合 |

> 兩者角色不衝突：n8n 是「讓流程自己跑」，LangSmith 是「看流程跑得對不對」。可並行、可只做其一。

---

## 軌道 A：n8n（優先，低風險）

**為什麼先做**：它早就在 [工具棧索引](../tools/工具棧索引.md) 的「觀望中」名單（「排程需求小，Jwood 自動化擴大時納入」），現在 Jwood/双云自動化正是那個觸發點；且開源可攜，完全不破壞 jos。

### 階段 1 — 起灶（第 1 週）
- [ ] 在現有 Zeabur 上自架一個 n8n 實例（Docker 一鍵；與双云平台同部署層，運維心智一致）。
- [ ] 接通第一條最小工作流驗證可用：挑一個**真實但低風險**的 Jwood 例子，如「新影片發布 → 自動產 IG 貼文草稿 → 通知」。
- [ ] 確認 n8n 能呼叫 Claude / 既有 API（這是它跟你 stack 接合的關鍵點）。

### 階段 2 — 試點（第 2–3 週）
- [ ] 把 1 條 Jwood 既有手動 SOP 整條搬上 n8n（建議：IG 每日貼文 SOP 或 YouTube 發布後的下游動作）。
- [ ] 跟「現在用 Claude Code / 手動」對比：**省了多少手動切換、漏錯率、是否真的更穩**。
- [ ] 記錄踩坑點（給 education 域當素材）。

### 階段 3 — 判定（第 4 週）
- **✅ 升格** → 寫進 [工具棧索引](../tools/工具棧索引.md) 主表（工具層）、`/移機 capture` 把 n8n 納入 [jos/environment](../../jos/environment/軟體清單.md) 清單、自架設定寫進 [自建程式清單](../../jos/environment/自建程式清單.md)。
- **❌ 關檔** → 移到本表「已關檔」，寫清楚為什麼（避免半年後重試）。

**升格條件**：連續 2 週承擔 ≥1 條真實 Jwood/双云流程且零重大故障，且省下的手動成本明顯 > n8n 維護成本。

---

## 軌道 B：LangSmith（條件性，看双云客戶量）

**為什麼擺後面**：它解的是「双云腦+手平台規模化後的黑盒子」痛點。**此痛點尚未明確爆發前不投**——避免為了還沒發生的問題養一套 SaaS。

### 啟動門檻（任一觸發才進場）
- 双云平台客戶量 / 每日 agent 執行量上到「人工已追不動 trace」。
- 出現 ≥2 次「客戶端結果不對、但查不出腦或手哪一步出錯」的事故。
- 要對客戶證明「AI 行銷部每天做了什麼、決策依據是什麼」的稽核需求。

### 階段 1 — 探接合成本（觸發後第 1 週）
- [ ] 確認 LangSmith 對**非 LangChain 架構**（你的 monorepo 腦+手）的串接方式與成本——這是最大不確定點，先驗證再投入。
- [ ] 只接 1 個品牌客戶的 pipeline 做 trace 試點。

### 階段 2 — 判定
- **✅** → 接合成本可接受 → 寫進工具棧索引（觀測層），並在 [双云 Agent 平台](../../portfolio/projects/双云agent平台.md) 補觀測層說明。
- **❌**（接合成本太高 / 黑盒子問題用自建 log 就能解）→ 關檔，改用自建 trace 方案。

---

## 不採用的五個（落地版理由，存查）

| 平台 | 不採用的一句話理由 |
|---|---|
| **CrewAI** | 你的双云 monorepo 腦+手 + AGENTS® 就是你自己的多 agent 框架，換它＝拆自建核心資產 |
| **Cloud Run** | Zeabur 已在 jos 部署層服役，換它＝抬高運維 + 綁 GCP |
| **Vertex AI Agent** | 綁 Google/Gemini，與 Anthropic-first + jos 廠商中立正面對撞 |
| **Bedrock Agents** | 只對已深耕 AWS 的團隊划算，你沒有 AWS 包袱 |
| **Zapier Agents** | SaaS 廠商鎖定 + 跟你 MCP 重疊 + 換機回不來；有 n8n 即不需要 |

> 例外條款：若未來接到「指定 GCP / AWS」的企業客戶，再就該案單獨評估對應雲平台——不改變自身 OS 預設。

---

## 落地檢查點（一眼看進度）

- [ ] **A1** n8n 在 Zeabur 起灶、跑通第一條最小工作流
- [ ] **A2** 1 條真實 Jwood SOP 搬上 n8n 並對比
- [ ] **A3** n8n 判定（升格寫進 jos / 關檔）
- [ ] **B0** LangSmith 啟動門檻是否觸發（未觸發＝正常，不投）
- [ ] **B1** 觸發後先驗非 LangChain 接合成本
