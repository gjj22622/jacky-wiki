# 高併發活動撐流量設計法（單 Node LLM 串流服務）

> 一個**單一 Node process、零依賴 http、串流接 LLM**的服務，在「實體活動短時間湧入 ~100 人」前，怎麼確認撐得住、怎麼補。
> 來源：TBSA Agents 平台（agents.tbsa.tw）6/26 說明會（預估 100 人）上線前壓力盤點。
> 配對：token 併發策略見 [admin-token-backend 規格書](admin-token-backend-規格書.md)；env 改要重 deploy 見 [Zeabur 自動部署方法](zeabur自動部署方法.md)。

---

## 🧠 核心認知：扛不住的不是「人數」，是四個地雷

單 Node 事件迴圈處理「瀏覽＋登入＋LLM 串流」這種 **I/O-bound** 工作，幾百條並發連線輕鬆。真正會炸的是下面四件事——**而且其中三件是「實體活動專屬」**，平常測不出來：

| # | 地雷 | 為什麼平常測不到 | 解 |
|---|---|---|---|
| 1 | **請求熱路徑做同步檔案 I/O** | 低流量時同步讀寫整檔只是 ms 級 | 記憶體為真相＋定時非同步批次落地 |
| 2 | **按 IP 的限流，遇到「全場共用一條 WiFi」** | 平常每人不同 IP，測不出來 | 活動當天大幅放寬限流 env |
| 3 | **共用一把 token＋"kick" 併發策略 → 互踢** | 平常每人一張 token | 共用 token 必須關掉併發上限 |
| 4 | **對上游 LLM 無限併發 → provider 429** | 平常沒有瞬間爆量 | 生成併發閘＋排隊優雅降級 |

> 心法：**「同時數」≠「人數」**。100 人不會在同一秒一起按生成（要讀、想、打字），真正同時打上游的大概 10~40。所有上限都該照「同時數」設，而且**做成 env 可一秒微調**，不要寫死。

---

## 地雷 1：請求熱路徑禁同步 fs

埋點/記錄類功能最容易中。原本「偶爾才寫」的 log，一旦接到**每次點擊**（公開頁、每顆按鈕）就變高頻；若它是「同步讀整檔 JSON → push → 同步寫整檔」（還 pretty-print），就會卡死單執行緒事件迴圈，**全站**跟著慢（連靜態檔、串流都排在後面）。

**解**：記憶體陣列當單一真相來源，寫入只更新記憶體＋標記 dirty，背景 `setInterval` 每 3 秒用 `fs.writeFile`(async) 批次落地；`SIGTERM`/`SIGINT` 先 flush 再 `process.exit(0)`（**加了 SIGTERM 監聽會蓋掉預設結束行為，必須自己 exit**）。讀取一律走記憶體。容忍 crash 丟最近 ≤3 秒事件（分析類可接受）。

---

## 地雷 2：會場共用 IP 把限流打爆（最容易被忽略、最致命）

`loginRateAllowed`、beacon 限流這類**按 IP** 的防護，在會場「100 人共用一條 WiFi＝同一個對外公網 IP」時，會把**整個會場**當成一個狂打的攻擊者擋掉。預設 `LOGIN_RATE_PER_MIN=10` → 一分鐘第 11 個登入就「嘗試太頻繁」→ 現場幾乎沒人進得去。

**解**：活動當天把限流 env 大幅放寬，活動後調回：

```
LOGIN_RATE_PER_MIN = 300   # 原 10
BEACON_PER_MIN     = 600   # 原 120（beacon 掉資料不影響 UX，次要但一併放寬）
```

> 進階：若能拿到會場 IP，可做白名單豁免；但「臨時放寬 env」最快、最不易出錯。

---

## 地雷 3：共用 token 的併發策略會互踢

活動發**一把共用體驗 token** 給全場最省事，但若該 token 設了 `maxConcurrent` 又用 `duplicateLoginPolicy:"kick"`，第 2 個人登入就把第 1 個踢掉 → 永遠只有 N 人在線。

**解**：共用 token 必須 `maxConcurrent=null`（不限）＋ `usesRemaining=null`（不限）。這樣併發檢查直接放行、不互踢。要更乾淨就一人一張（但 100 張要先生成）。詳見 [admin-token-backend 規格書](admin-token-backend-規格書.md)。

## 地雷 4：生成併發閘＋排隊（對上游 LLM 優雅降級）

app 對上游（OpenRouter 等）若無併發上限，100 人同時按生成＝100 條串流瞬間砸過去，超過 provider 速率就 429，使用者看到醜錯誤。**Node 扛得住 100 條串流，但 provider 不一定。**

**解**：一個輕量併發閘——同時最多 N 條，超過排隊；佇列滿或等太久回友善 503，導去「複製提示詞自己跑」（把流量卸載到使用者自己的 ChatGPT）。使用者中途離開即 abort，把名額讓給下一位。

```js
// 生成併發閘：N＝同時生成數（不是人數！100 人活動設 20~30），全做成 env 可調
const GEN_MAX_CONCURRENT = Number(process.env.GEN_MAX_CONCURRENT || 24);
const GEN_QUEUE_MAX      = Number(process.env.GEN_QUEUE_MAX || 200);
const GEN_QUEUE_WAIT_MS  = Number(process.env.GEN_QUEUE_WAIT_MS || 25000);
let genActive = 0;
const genQueue = [];
function acquireGenSlot() {
  return new Promise((resolve) => {
    if (genActive < GEN_MAX_CONCURRENT) { genActive += 1; return resolve({ ok: true }); }
    if (genQueue.length >= GEN_QUEUE_MAX) return resolve({ ok: false, reason: "FULL" });
    const w = { resolve, timer: null };
    w.timer = setTimeout(() => {
      const i = genQueue.indexOf(w); if (i >= 0) genQueue.splice(i, 1);
      resolve({ ok: false, reason: "TIMEOUT" });
    }, GEN_QUEUE_WAIT_MS);
    genQueue.push(w);
  });
}
function releaseGenSlot() {
  const next = genQueue.shift();
  if (next) { clearTimeout(next.timer); next.resolve({ ok: true }); } // 空位直接交棒，genActive 不變
  else if (genActive > 0) genActive -= 1;
}

// handler 內：驗證(session/權限/KEY)通過「之後」才取位（別為 4xx 佔位）
async function handle(req, res) {
  // ...session / permission / input / KEY 檢查...
  const slot = await acquireGenSlot();
  if (!slot.ok) return sendJson(res, 503, { error: "BUSY", fallback: GEN_BUSY_MSG });
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 180000);
  req.on("close", () => { try { controller.abort(); } catch {} }); // 離開即中止、騰名額
  try {
    // ...fetch(stream) + 串流回寫；早退(502)也 OK，finally 會釋放...
  } catch (e) { /* ... */ }
  finally { clearTimeout(timer); releaseGenSlot(); } // finally 每個 try 跑一次，含早退路徑
}
```

**關鍵正確性點**：
- 取位放在所有便宜驗證**之後**（不要為 400/401 佔用名額）。
- `finally` 釋放 → 涵蓋成功、錯誤、早退(`return` 在 try 內也會走 finally)。
- 釋放時若有排隊者，**直接交棒、`genActive` 不變**（不是先 -1 再 +1，避免瞬間缺口）。
- `req.on("close")` 只 `abort`，由 `finally` 統一釋放（idempotent，不重複扣）。
- 邏輯可離線單元測試：兩個佔滿→第 3 個排隊→佇列滿回 FULL→釋放一個交棒(genActive 不變)→逾時回 TIMEOUT→全釋放不會負數。

## ✅ 上線前檢查清單（活動前一天）

- [ ] **熱路徑無同步 fs**（埋點/log 走記憶體＋非同步落地）
- [ ] **放寬按 IP 限流** env（會場共用 WiFi）：`LOGIN_RATE_PER_MIN`、`BEACON_PER_MIN`
- [ ] **共用 token** `maxConcurrent=null`＋`usesRemaining=null`（不互踢、不限次）
- [ ] **生成併發閘** N 設 20~30（`GEN_MAX_CONCURRENT`），確認排隊降級訊息會顯示
- [ ] **上游 provider**：確認 rate limit／併發上限／餘額（這是唯一程式無法代償的外部瓶頸）
- [ ] **容器規格**：單 process 無自動擴容 → 當天暫時加 CPU/RAM
- [ ] **env 改完重新 deploy**（restart 不重載 env，見 Zeabur 鐵則 2）
- [ ] **節流壓測**：本機或 staging，**別對正式站灌**（對 prod 灌併發會被安全機制/良知擋下，也會干擾真實使用者）

## 採坑

- 對「已經很慢的正式站」灌 40+ 併發做壓測 → 被擋（合理）。要測就本機/staging 節流測，prod 只做單發 smoke。
- 設 env 後沒重 deploy → 改了等於沒改（restart 用舊快照）。
- `npx zeabur variable create` 偶發 TLS x509 unknown authority → transient，重試即可（deploy 走另一路徑通常不受影響）。
- Git Bash 顯示 `curl -w` label 把 `/admin`、`/18week/` 等字串轉成 `C:/Program Files/Git/...` 是假象，看 HTTP 碼即可。
