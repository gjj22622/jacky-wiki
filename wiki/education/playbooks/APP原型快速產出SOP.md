# APP 原型快速產出 SOP

> 把「客戶想做 APP」從「紙本規格」變成「客戶簽認的可點原型」——**整套不寫任何 RN 程式碼**，全部用 HTML 模擬，1 週內完成。
>
> 來源：J董 Labor APP（人力仲介數位管理）2026-05-19 實戰。配對：[客製 APP 接案服務化方法論](客製APP接案服務化方法論.md)（商業面）。

---

## 適用情境

| 對 | 不對 |
|---|---|
| 客戶手上只有紙本/txt 規格，想做 mobile APP | 客戶已有完整 spec + RN 開發團隊只缺工程實作 |
| 預算/技術路線未定，要先簽認方向 | 純技術 PoC 任務（不需客戶端展示） |
| 1-2 週內要拿到客戶決策 | 給 IT 內部用、不需客戶簽認的工具 |
| 客戶決策者願意坐下開 90 分鐘會 | 客戶只能 email 來回、無法當面跑流程 |

## 前置條件

開工前確認以下到手：

- ☐ 客戶基礎規格文件（任何格式：紙本拍照、txt、word、Notion 連結）
- ☐ 客戶可派 1 位決策者面對面開 90 分鐘會議
- ☐ Claude Code（Opus 4.7 建議，1M context 必要）+ Web 瀏覽器
- ☐ Git 環境 + 一個空白資料夾當專案根
- ☐ 客戶端電腦能開 Chrome/Edge/Safari（不需安裝任何軟體）

## 7 階段步驟

### 階段 1 — 進駐與規格消化（30 分鐘）

- [ ] **1.1** 建立專案資料夾，把客戶規格存進去（任何格式皆可）
- [ ] **1.2** 用 `Read` 工具讀完規格，AI 自動萃取 5 個重點：
  - 核心目標一句話
  - 必填欄位 / 必要流程
  - 技術需求 / 部署需求
  - 隱含但沒明寫的需求
  - 跟客戶業內常識牴觸的設計
- [ ] **1.3** 產出「現況盤點」一頁式回覆給 J董/主導者確認方向
- [ ] **1.4** 此階段**不要動工寫任何程式**，先對齊認知

**為什麼**：80% 客戶規格漏掉的隱含需求，現在挑出來比 MVP 階段才發現便宜 10 倍。

---

### 階段 2 — 三路平行 agent 市場掃描（25 分鐘 wall clock）

- [ ] **2.1** 同時派 3 個 agent（在同一個 message 內 3 個 Agent tool call）：
  - **Agent A**：產業現況 + 法規 + 政府平台（用 WebSearch + WebFetch）
  - **Agent B**：本土競品掃描 + 功能對照 + 定價錨點
  - **Agent C**：GitHub 開源類似專案 + 技術選型建議
- [ ] **2.2** 每個 agent 給 1200-2000 字繁中報告 + 最後一段「對客戶規格的啟示」
- [ ] **2.3** 合成「市場調查報告.md」+ 三色 callout（綠燈時機 / 黃燈警報 / 紅燈風險）高層 briefing

**鐵律**：主對話想清楚，sub-agent 做乾淨。三個 agent 的 prompt 都要**自成一體**——不能要 agent 「看本對話」。

> 延伸閱讀：[Commander+Executor 單人多 Agent 模式](../../cross-domain/Commander+Executor單人多Agent模式.md)

---

### 階段 3 — 導覽頁 + 會議回填表單（半天）

- [ ] **3.1** 寫一份 `index.html` 整合三件事：
  - 左側 sticky 目錄（11+ 錨點）
  - 中段：市場調查 + 競品對照 + 解法 + 開源評估
  - 末段：**會議回填表單**（15 題決策節點）
- [ ] **3.2** 表單功能：
  - 必拍板紅標 / 重要黃標 / 加分綠標
  - 推薦選項加綠底「推薦」標籤
  - `localStorage` 每 30 秒自動暫存
  - 按「產生會議結論」匯出 Markdown
  - 「複製到剪貼簿」/「下載 .md」/「暫存草稿」/「載入草稿」
- [ ] **3.3** 設計重點：
  - **亮色版**（深色版客戶會看不清）
  - 無外部依賴（單檔 HTML）
  - 桌機 + 行動裝置 RWD

> 延伸閱讀：[入口導覽頁 SOP](../../cross-domain/入口導覽頁SOP.md)

**採坑點**：不要做兩個獨立檔（導覽 + 表單），整合進同一檔，客戶不會在多 tab 之間切。

---

### 階段 4 — 客戶會議與結論回填（會議當天 + 30 分鐘整理）

- [ ] **4.1** J董/主導者帶 `index.html` 去開會（離線可開，但建議線上跑以便 localStorage 暫存）
- [ ] **4.2** 會議過程中逐題勾選決策，重點：
  - 不要追求 100% 答完，未填的會自動標「待確認」
  - 自由筆記區記客戶金句
  - 客戶答應提供的資料勾起來
- [ ] **4.3** 會議結束按「產生會議結論」複製 Markdown 給技術端
- [ ] **4.4** 技術端存成 `會議結論_YYYY-MM-DD.md`，並標示**衝突警報**：
  - 預算 vs 時程 vs 範圍 三角衝突
  - 業務上下文牴觸的功能（如離線優先 vs 純雲端）
  - 未來會卡的監管風險

**採坑點**：客戶當下說的「全部都要」+「3 個月內」+「30 萬以下」常常數學不成立——衝突要在會議結論裡白紙黑字標示，但**不要在客戶面前直接打臉**。

---

### 階段 5 — MVP 互動原型（1-2 天）

- [ ] **5.1** 寫 `mvp.html`，**手機框 + 5-7 個核心畫面**：
  - iPhone 樣式黑框 + 圓角 + notch
  - 380×780 桌面預覽 / 手機自動全螢幕（RWD）
  - 底部 tab bar 4 顆按鈕（主畫面 + 副功能）
  - 左右切換 / 點 case 進詳情 / 抽屜瀏覽
- [ ] **5.2** 核心畫面清單（依案件性質調整）：
  - 登入
  - 主功能列表
  - 詳情頁 + 分類抽屜
  - 抽屜內檔案
  - 拍照 / 表單 / 新建流程
  - 主動提醒清單
  - 我的（含登出）
- [ ] **5.3** **「拍照開案」OCR 模擬流程**（最能感動客戶的功能）：
  - 模式選擇：📷 拍照開案 vs ✍️ 手動輸入
  - 拍照頁 → 掃描動畫（5-6 步）→ 確認預填表單 → 建立
  - 用 3-4 組假 OCR 樣本隨機抽，每次拍都不同雇主資料
- [ ] **5.4** 假資料 + localStorage 持久化（key 命名加版本：`projectname_state_v1`）
- [ ] **5.5** **接客戶真實案例**（若有）：
  - 把 5-10 份真實 PDF 放 `真實案例/` 資料夾
  - 抽屜內檔案項目用 `<a target="_blank">` 點擊可開原始 PDF
  - 業務規則 chip 標籤（如 Labor 案的「雇主 / 受照顧者 / 移工」三色標籤）

**採坑點**：
- 登入頁 inline `style="display:flex"` 會蓋掉 `.page.active` 的切換邏輯 → **任何 page 不要用 inline display style**
- localStorage key 沒升版 → 改 schema 後客戶看到舊資料 → 每次資料模型改就升 `_v1` → `_v2`
- 「自動歸位」這種抽象規則用文字講客戶聽不懂，用**彩色 chip 標籤**一秒懂

---

### 階段 6 — 教育訓練 + 客戶交付清單（半天）

- [ ] **6.1** 寫 `training.html`，14 張投影片，5 個 Part：
  - Part 1 系統介紹：定位、五層架構（行動端/後端/雲端/AI/資安）、6 大資料結構
  - Part 2 運作流程：拍照開案 5 步、既有業務拍照 3 步、主動提醒推播
  - Part 3 三角色教育訓練（依客戶實際角色配置）
  - Part 4 客戶交付清單：4 類資料 + 15-20 項**可勾選檢核表**（localStorage 記憶 + 一鍵匯出）
  - Part 5 Q&A 6 個常見問題 + 6 個里程碑時間軸
- [ ] **6.2** 簡報 UX：
  - scroll-snap 切頁
  - 鍵盤 ↑↓ / PageDn / Home / End / F 全螢幕
  - `@media print` 列印友善（每張一頁）
- [ ] **6.3** 客戶交付清單可一鍵匯出：
  - `navigator.clipboard.writeText()` 複製 Markdown
  - 已勾 vs 未勾分組
  - 客戶貼 LINE/Email 回傳即可

---

### 階段 7 — 客戶交付包打包 + Git 版控

- [ ] **7.1** 建 `客戶交付包/` 資料夾，複製：
  - `pitch.html`（提案簡報，若有獨立做）
  - `mvp.html`
  - `training.html`
  - `真實案例/` 整個資料夾
- [ ] **7.2** 寫 `客戶交付包/導覽.html`：
  - **客戶版簡化入口**（不含內部用的會議回填表單）
  - 3 張卡片：①提案 ②MVP 體驗 ③系統架構+交付清單
  - 建議瀏覽順序明確
- [ ] **7.3** 寫 `客戶交付包/使用說明.txt`：純文字，IT 阿姨阿伯都看得懂
- [ ] **7.4** Git 分支策略：
  - 每個產出走獨立 feature branch（`pitch-deck` / `mvp-demo` / `training-deck`）
  - `merge --no-ff` 保留分支歷史線
  - commit 格式：`feat(<scope>): 描述` + Co-Authored-By（依 [全域規範](~/CLAUDE.md)）
- [ ] **7.5** 客戶若要 zip 寄出 → PowerShell `Compress-Archive` 即可

**採坑點**：客戶交付包不要直接複製內部用的 `index.html`（含會議回填表單客戶用不到）——要做簡化版。

## 異常情況分支

| 情況 | 處理 |
|---|---|
| 客戶規格太薄 | 階段 1 不寫推測，直接列 5 個待釐清問題回覆主導者 |
| 三路 agent 中有 agent 找不到資料 | 不要硬編造，誠實標「市場資訊稀缺」並推測原因 |
| 會議當天客戶答案矛盾 | 在會議結論裡明確標衝突警報，**不要在客戶面前打臉** |
| MVP 寫到一半發現業務邏輯有變 | 開新 feature 分支處理，避免污染現有 main |
| 客戶說「視覺先不用考慮」 | system font + 1 種主色 + 大量留白，**絕不**做精美視覺白費 token |
| 客戶手機分布不明 | 預設做 RN（雙平台），若客戶 100% Android 改 Expo Web → APK 也可 |
| 預算金額由主導者（J董）處理 | 我方產出範圍/時程/工時，**不主動報價** |

## 採坑點集合

| # | 採坑 | 解法 |
|---|---|---|
| 1 | 登入頁 inline `display:flex` 蓋切換邏輯 | 任何 page 不要用 inline display style |
| 2 | localStorage key 沒升版 | schema 改就升 v1→v2，邊改邊測 |
| 3 | 預算/報價自作主張寫進文件 | 主導者要的是範圍時程，不是金額 |
| 4 | 6 抽屜結構沒考慮「文件主體歸屬」 | MVP 階段一定要把業務 chip 標籤露出來，客戶看到才會校正 |
| 5 | 客戶交付包混入內部用文件 | 簡化版入口頁 + 純文字使用說明 |
| 6 | 三路 agent prompt 不夠自成一體 | Agent 看不到本對話，所有 context 都要塞進 prompt |
| 7 | mvp.html 一檔太長改不動 | CSS 系統先抽 root vars，HTML 用大塊 section 註解區分 |

## 驗收標準

- ✅ 客戶能用 `index.html` 自己預習提案（不需我方陪同）
- ✅ 客戶能在 `mvp.html` 親手點按一輪 5 個核心流程
- ✅ 客戶能在 `training.html` 第 12 頁勾選後一鍵匯出交付清單
- ✅ 客戶交付包資料夾打包寄出後客戶能自己點開所有檔案
- ✅ 客戶簽認 MVP 後內部團隊可接續開發（spec 凍結）
- ✅ Git 歷史線清晰，每個 feature 分支可獨立回溯

## 快速回顧

**5 工作天從紙本規格到客戶簽認可點原型**，全程不寫真實 RN 程式碼。三路平行 agent 縮短市場研究、HTML 模擬手機 APP 換取速度、會議回填表單把客戶決策結構化、客戶交付包讓客戶能自己消化——這套流程的精神是「**用最低成本的可點實物換最高信任的客戶決策**」。
