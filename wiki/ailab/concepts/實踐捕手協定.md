# 實踐捕手協定 ⭐

> **目的**：讓 Jacky 在**任何 AI 模型／任何對話**下，都能用同一份格式把 AI 實踐扎實記錄下來。
> **設計原則**：協定本身**模型無關**——Claude Code、Codex CLI、ChatGPT、Gemini、未來新模型，都能輸出符合本協定的事件紀錄。

---

## 兩種捕捉模式

| 模式 | 何時用 | 結構 | 觸發 |
|---|---|---|---|
| **Mode A：單一事件**（v1.0）| 當下發現「這個值得記」（單一突破／失敗／工具）| 9 欄位 | `/ailab capture <一句話>` |
| **Mode B：Session 總結**（v1.0）| **對話結尾或中間斷點**——回顧整段工作 | 5 區塊 | `/ailab session [<主題>]` |

> **判斷準則**：腦中是「**一個事件**」用 A；「**這段對話**」用 B。

---

## 為什麼要有協定（不只是 prompt）

| 問題 | 沒協定的後果 | 有協定的好處 |
|---|---|---|
| 跨模型 | Claude Code 有 auto-memory，ChatGPT 沒有，Gemini 各自獨立 | 統一格式 → 任何模型輸出都能直接 ingest |
| 跨對話 | 每次對話從零開始描述 | 9 個固定欄位 → 不漏關鍵維度 |
| 跨時間 | 兩個月後看不懂自己當時記什麼 | 結構化 → 時序、模型、成熟度都能追 |
| 升格難 | 流水帳混在工作日誌 → 升格 wiki 時無法判斷 | 自帶 maturity 與升格目標 → 漏斗判斷直接化 |

## Mode A：單一事件 — 9 個欄位

### 模板（複製這段給任何 AI）

```
---
event: <一句話描述發生了什麼>
date: YYYY-MM-DD
model: <claude-opus-4.7 / claude-sonnet-4.6 / gpt-5.4 / gemini-2.5-pro / ...>
context: <在哪個專案／場景發生（双云／Jwood／NCHU／個人實驗 / ...）>
type: <突破 | 失敗 | 工具發現 | 模式發現 | 模型對比 | 工作流改進>
maturity: <想法 | 實驗中 | 已驗證 | 已穩定>
tags: <自由標籤，用逗號分隔，如：mcp, prompt-cache, codex>
---

## 發生了什麼
（1-3 句直白描述）

## 為什麼重要
（為什麼這個事件值得記？跟之前比有什麼不同？）

## 怎麼做的
（操作步驟、具體 prompt、工具設定、命令、設定檔片段）

## 對比與替代
（之前怎麼做？為什麼換？有別的選項嗎？）

## 連結與出處
（相關 wiki 頁、外部資料、commit hash、檔案路徑）

## 升格目標（建議落點）
（建議放到 ailab/<concepts|tools|patterns|experiments|log>/<檔名>，或暫留 inbox）
```

### 欄位定義

| 欄位 | 必填 | 說明 |
|---|:---:|---|
| `event` | ✅ | 一句話描述（不要超過 30 字） |
| `date` | ✅ | YYYY-MM-DD（事件發生日，不是記錄日） |
| `model` | ✅ | 用了哪個模型（含版號）。多模型對比就用逗號列出 |
| `context` | ✅ | 哪個專案／場景。如：`双云客戶X`、`Jwood AI 工作流`、`NCHU 第三次專討`、`個人實驗` |
| `type` | ✅ | 6 種固定值之一（見下方） |
| `maturity` | ✅ | 4 種固定值之一（見下方） |
| `tags` | ⭕ | 自由標籤，方便日後 grep |
| 五段內文 | ✅ | 「發生了什麼／為什麼重要／怎麼做的／對比與替代／連結」必寫；「升格目標」可選 |

### `type` 六種值

| 值 | 用於 |
|---|---|
| `突破` | 解開了卡住的問題、找到新做法 |
| `失敗` | 試了不通、踩到坑（如 73% 幻覺事故）|
| `工具發現` | 發現新 MCP / 新 Skill / 新 CLI / 新功能 |
| `模式發現` | 多次重複的做法被察覺成模式（候選 patterns/）|
| `模型對比` | 同任務在不同模型／不同版本上的差異 |
| `工作流改進` | 既有流程的改良（不是新發現是優化）|

### `maturity` 四種值

| 值 | 定義 | 落點建議 |
|---|---|---|
| `想法` | 只是有個念頭，還沒做 | inbox/ |
| `實驗中` | 做了但還在驗證 | experiments/ |
| `已驗證` | 至少成功跑過一次 | experiments/ 或 log/ |
| `已穩定` | 重複用過、確定可靠 | concepts/ / tools/ / patterns/ |

## Mode B：Session 總結 — 5 區塊（對話結尾／中間斷點用）

### 為什麼要有 Mode B

對話進行 1-2 小時、跑完一個任務後，腦中累積的不是「一個事件」，是**一連串混在一起的事**：
- ✅ 試過幾種做法，**最後**走通的那條
- 🟡 試過但**走錯路**（繞路）的那幾條
- ❌ 直接撞牆的**錯誤**

如果用 Mode A 拆成 5 個獨立 capture 太繁瑣，且**繞路與錯誤的價值在於「跟最終做法的對照」**——分開存反而失去脈絡。

→ Mode B 是「**對話即一篇日記**」的格式，一份檔案保留完整脈絡。

### 模板（複製這段給任何 AI）

```
---
session_date: YYYY-MM-DD
session_topic: <一句話描述這次對話的主題＞
model: <主用模型，多模型用逗號>
context: <專案／場景>
duration: <大致時間，如 ~2h，可選>
type: session-summary
tags: <逗號分隔>
---

## 最終做法（What Worked）
- 具體做法、prompt、工具設定、命令
- 為什麼這做法穩
（這段是「**結論**」——下次直接照這做的依據）

## 繞路紀錄（Detours）
- 試 A 不通 → 因為 X → 改試 B
- 試 B 半成 → 因為 Y → 改試 C
（這段最珍貴——下次想到「要不要試 A」時來查，省半天）

## 錯誤與失敗（What Failed）
- 試 X 直接撞牆 → 原因
- 一開始假設錯了 <某事> → 後來修正成 <什麼>
（這段是「**反省**」——同類錯誤下次別再犯）

## 升格候選
- 哪些值得升 ailab/tools/<檔> 或 patterns/<檔>
- 哪些只留 inbox 不升
- 哪些是長期實驗（候選 experiments/）

## 待延伸（Next）
- 下次對話要做什麼
- 還沒驗證的假設
- 觸發後續 capture 的事件
```

### 5 區塊定義

| 區塊 | 必寫 | 說明 |
|---|:---:|---|
| 最終做法 | ✅ | 下次直接拿來用的「**結論**」 |
| 繞路紀錄 | ⭕ | 沒繞路就不寫；有就寫——**繞路的對照價值最高** |
| 錯誤與失敗 | ⭕ | 沒錯誤就不寫；有就寫含原因 |
| 升格候選 | ✅ | 強迫做漏斗判斷——避免事件死在 inbox |
| 待延伸 | ⭕ | 沒就空著 |

### `type: session-summary` 的特殊處理

- **不會自動升格**：Mode B 是脈絡保留，不是事件本身
- 處理方式：升格時拆解成多個 Mode A 事件後再升，原 Mode B 留 inbox 當歷史
- 例：一份 session-summary 含「Codex 比 Claude 穩」這個發現 → 升格時把這條抽出來、用 Mode A 寫成獨立檔升 `tools/模型選擇心法.md`

## 三種使用情境

### 情境 1：在 Claude Code 內（最方便）

**Mode A（單一事件）**：

```
/ailab capture <一句話描述>
```

或在對話中說：「**這個值得記。請用實踐捕手協定捕一份。**」

**Mode B（對話結尾／中間斷點）**：

```
/ailab session [<主題>]
```

或自然語言：「**幫我用 ailab session 模式總結這次對話。**」

→ Claude Code 會自動讀 `~/.claude/skills/ailab/SKILL.md`，掃過對話脈絡，套用 Mode A 或 Mode B 格式，寫到：
- Mode A：`wiki/ailab/inbox/<日期>-<event-slug>.md`
- Mode B：`wiki/ailab/inbox/<日期>-session-<topic-slug>.md`

### 情境 2：在其他 AI 對話（ChatGPT／Codex CLI／Gemini／Claude Web）

#### Mode A prompt（單一事件）

```
我要用「實踐捕手協定 v1.0 Mode A」記錄剛才這個 AI 實踐事件。
請用以下 9 欄位格式輸出（其他都不要說）：

---
event: <30 字內>
date: <YYYY-MM-DD>
model: <你是哪個模型>
context: <我用在哪個專案>
type: <突破|失敗|工具發現|模式發現|模型對比|工作流改進>
maturity: <想法|實驗中|已驗證|已穩定>
tags: <逗號分隔>
---

## 發生了什麼
## 為什麼重要
## 怎麼做的
## 對比與替代
## 連結與出處
## 升格目標
```

#### Mode B prompt（對話結尾／中間斷點）

```
我們這次對話準備告一段落（或要切換主題）。
請用「實踐捕手協定 v1.0 Mode B」幫我總結這次對話的 AI 實踐紀錄，
重點要保留「最終做法／繞路／錯誤」三方對照——其他都不要說：

---
session_date: <YYYY-MM-DD>
session_topic: <一句話描述這次對話主題>
model: <你是哪個模型>
context: <用在哪個專案>
duration: <大致時間>
type: session-summary
tags: <逗號分隔>
---

## 最終做法（What Worked）
（具體做法、prompt、命令——下次照這做）

## 繞路紀錄（Detours）
（試 A 不通因為 X → 試 B 半成因為 Y → 試 C 通了）

## 錯誤與失敗（What Failed）
（試 X 撞牆原因；一開始假設錯了什麼）

## 升格候選
（哪些值得升 ailab/tools 或 patterns，哪些只留 inbox）

## 待延伸（Next）
（下次要做什麼、未驗證假設）
```

→ AI 輸出後，**Jacky 複製貼到** `OneDrive/ailab-inbox/<日期>-<event-or-session>.md`（後續批次處理）。

### 情境 3：手機／無 AI 環境（最低門檻）

不寫完整協定，只用三行極簡格式：

```
@ailab YYYY-MM-DD <model> | <event>
why: <一句話為什麼重要>
how: <一句話怎麼做的>
```

→ 丟進 OneDrive `ailab-inbox/raw.md` 末尾，下次進 Claude Code 用 `/ailab promote` 自動補完協定 9 欄位後升格。

## 三種 inbox 路徑

| 來源 | 落點 | 說明 |
|---|---|---|
| Claude Code 即時記 | `wiki/ailab/inbox/<日期>-<event>.md` | 完整協定格式 |
| 其他 AI 跨對話 | `OneDrive/ailab-inbox/<日期>-<event>.md` | 完整協定格式（手動貼）|
| 極簡手機記 | `OneDrive/ailab-inbox/raw.md` | 三行極簡，事後補完整 |

> Claude Code 的 inbox 在 wiki 內方便 commit；OneDrive inbox 處理「Claude Code 不在身邊」的場景，本機之外的事件先丟那裡。

## 升格流程（依事件、不依時間）

```
[事件發生]
   ↓
[用協定捕] (任何 AI／任何對話／任何裝置)
   ↓
[落 inbox] (wiki/ailab/inbox/ 或 OneDrive/ailab-inbox/)
   ↓
─── 由「事件成熟度」判斷 ───
   ├─ 想法 / 實驗中 → 留 inbox 或 experiments/
   ├─ 已驗證 重複出現 → patterns/ 或 tools/
   └─ 已穩定 形成新方法 → concepts/ 或 log/（重大里程碑）
```

> 詳見 [三層萃取漏斗](三層萃取漏斗.md)。
> **無時間門檻**：不用機械化的 30 天 / 3 個月閘門，由事件本身的成熟度與重複度決定升格。

## 範例 1：突破（多模型對比）

```
---
event: Codex CLI gpt-5.4 比 Claude Sonnet 4.6 更會處理跨檔重構
date: 2026-04-25
model: gpt-5.4, claude-sonnet-4.6
context: Jwood Brand System 重構
type: 模型對比
maturity: 已驗證
tags: codex, refactor, multi-file, model-comparison
---

## 發生了什麼
同一個跨檔重構任務（5 個檔、共 800 行），Codex CLI 一次到位；
Sonnet 4.6 漏改了 2 個 import 路徑，Opus 4.7 沒測。

## 為什麼重要
之前以為 Claude 全面領先，但跨檔結構性重構 Codex 確實更穩。
這是工作分流的依據（不是 Claude 全包）。

## 怎麼做的
- Codex CLI: `codex run "重構這 5 個檔..."` 直接給
- Sonnet 4.6: 開 Claude Code，給同樣 prompt，逐檔 Edit

## 對比與替代
之前都用 Claude。這次因為 Sonnet 漏改，順手跑 Codex 對照才發現。

## 連結與出處
- Jwood/Jwood_Brand_System/ commit abc1234
- 相關討論在 ailab/tools/codex-cli.md

## 升格目標
ailab/tools/模型選擇心法.md 補一條「跨檔結構重構 → Codex」
```

## 範例 4（Mode B）：對話結尾總結（建 ailab 域當天）

```
---
session_date: 2026-05-02
session_topic: 在 jacky-wiki 新建第 6 個內容域 ailab（AI 實踐）並建立跨模型實踐捕手協定
model: claude-opus-4.7
context: jacky-wiki ingest
duration: ~3h
type: session-summary
tags: wiki, ailab, capture-protocol, skill-design
---

## 最終做法（What Worked）
- 規劃階段先列 4 個 Wave（骨架／內容／反向連結／Skill）+ 6 個拍板項，集中問 Jacky 一次
- 拍板後一次到位執行，不要分批讓 Jacky 中斷
- 協定 + Skill 「**雙生**」設計：協定（markdown）模型無關放 wiki；Skill（Claude Code）自動讀協定
- inbox 雙路徑：wiki/ailab/inbox/（Claude Code 內）+ OneDrive/ailab-inbox/（跨模型／手機）
- 「依事件升格、無時間門檻」是 Jacky 偏好——比機械化 30 天閘門更貼近真實節奏

## 繞路紀錄（Detours）
- 一開始想用 cd 進 wiki repo 跑 git——Bash tool 在 Windows 路徑解析有 bug，改用 PowerShell + 絕對路徑就過
- 一開始把 ailab inbox 設為機械化 30 天門檻——Jacky 拍板「無時間門檻、依事件」，刪掉門檻邏輯
- 一開始 stash pop 撞 conflict——log.md 與 wiki主索引.md 雙頭，手動解 conflict marker 比硬 rebase 快

## 錯誤與失敗（What Failed）
- 域名「AI 學習路徑」一度寫進規劃——Jacky 隱性訂正成「AI 實踐」（強調動手非時間軸），如果照原名做會變成日記性域
- 第一次 wiki 主索引頁數沒對齊（92 vs 93）——後來 rebase 完發現 upstream 還加了 1 頁（多品牌獨立應用架構），改成 94
- v1.0 協定漏了「對話結尾總結」這種多事件混合場景，Jacky 立刻問了——v1.1 補上 Mode B

## 升格候選
- ⭐ Mode A + Mode B 兩種捕捉模式 → 升格本協定為 v1.1（已動手）
- 「Skill + 協定雙生」設計 → 升格 ailab/concepts/AI工具觀.md「整合鏈完整 > 單點最強」段補例子
- Windows 上 Bash vs PowerShell 路徑差異 → ailab/tools/claude-code.md 雷區段補一條
- inbox 雙路徑（wiki + OneDrive）邏輯 → 已在實踐捕手協定本頁，不用再升

## 待延伸（Next）
- OneDrive/ailab-inbox/ 資料夾 Jacky 首次使用時建立（待）
- Claude Code Hook：對話結束自動觸發 /ailab session（EXP-2026-Q2-04）
- 第一份真實 Mode B 事件（不是這個範例）開始累積
- 觀察一個月後，看 Mode A vs Mode B 比例，調整協定權重
```

## 升級協定（v1.1+ 候選改動）

未來 v1.1 候選：
- 加 `cost`：本次事件的 token / 美金成本
- 加 `time-spent`：事件耗時
- 加 `replicable`：別人能不能複製（boolean）
