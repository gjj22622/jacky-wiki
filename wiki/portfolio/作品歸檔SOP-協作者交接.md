# 作品歸檔 SOP（給協作者交接）

> 這份是把 Jacky 作品域的歸檔方法**打包成可交接的指令**，給 J董（或任何有自己 wiki 的協作者）在**自己的 Claude Code + 自己的 wiki** 跑同一套。
> J董 有自己的 wiki 管理（jdong-wiki），用這套在他那邊建他自己的 portfolio。

---

## 給 J董：整段複製貼進你的 Claude Code

```
任務：把我所有的「作品」掃描歸檔進我自己的 wiki（建一個 portfolio 作品域）。

【作品定義】符合任一即算作品：
1. 已部署 / 有網址（線上產品）
2. AI 生成的簡報 / 素材文件
3. 有導覽頁 / 有 HTML 檔 的成品

【掃描範圍】我的專案資料夾（本機 + OneDrive 都要掃），列出頂層後逐一判斷。

【深掘鐵則】資料夾越豐富越值得記。某夾若「子夾 >5 個」或「深度 >3 層」，
就派一個專屬 agent 深掘那一夾（深 2-3 層），把裡面的【多個子專案/作品】逐一揪出來——
大夾常是一個「專案家族」，內含多個獨立作品，別只看表層。

【執行方式】派 agent team 並行調查（read-only），每個 agent 查一批資料夾，
回報結構化座標：定位 / git remote / 前端網址 / admin 網址 / 部署平台 / DNS /
狀態 / 憑證檔是否存在。寫入集中由你（主控）做，agent 只調查不寫。

【🔒 金鑰紅線（最重要）】token / 密碼 / API key 的【值】絕不寫進 wiki 頁面，
只寫進一個 gitignored 的金庫檔（如 wiki/portfolio/.vault/作品金鑰.local.json），
並把 wiki/portfolio/.vault/ 加進 .gitignore。agent 回報憑證只報「路徑」不貼值。
頁面只記「帳號 → 見金庫」。完成後用 git check-ignore 驗證金庫沒進 git。

【產出結構】
wiki/portfolio/
├── 作品索引.md          # 分類目錄：A 線上產品 / B 互動成品 / C 客戶交付 /
│                        #           D 課程方法論 / E 工具；豐富大夾用「專案家族」section
├── projects/<名稱>.md   # 已部署/旗艦作品給完整頁（座標表 + 登入見金庫 + 一鍵示範腳本）
├── .vault/作品金鑰.local.json  # gitignored 金庫（token/密碼）
└── README.md            # 金庫機制 + 隱私紅線

【每個作品頁五區塊】① 一句定位 ② 部署座標表(資料夾/repo/前端/admin/DNS)
③ 登入(只寫帳號→見金庫) ④ 一鍵示範腳本(開哪個網址/登入哪/展示什麼) ⑤ 相關連結。

【交叉比對】同一作品的 OneDrive 工作副本 vs 本機 git 源，判斷是否同一個，避免重複歸檔。

先掃頂層列清單給我確認分批，再派 agent team。完成後 commit（feat 格式）但先別 push，給我看。
```

---

## 方法論精髓（為什麼這樣設計）

| 設計 | 為什麼 |
|---|---|
| **作品定義放寬** | 不只線上服務；AI 簡報、導覽頁、HTML 成品都是要拿來示範/檢閱的作品 |
| **深掘鐵則（>5 子夾 / >3 層）** | 表層掃會漏掉「專案家族」內的已部署子產品（實證：漏掉 4 個有網址的產品）|
| **agent team 並行 + 主控集中寫** | 並行快；寫入集中才守得住金鑰紅線 |
| **分離金庫（gitignored）** | wiki 是 git（可能 push public），token 絕不能進；但示範又要查得到 → 金庫解 |
| **指標式憑證** | agent 不貼值、頁面不存值，只記「憑證在哪個檔」+ admin 網址 |
| **分類目錄 + 專案家族** | 量大時，分類 + 家族巢狀比 N 個散頁更利於檢閱/示範 |
